\textbf{沙盒语料规模与检索接口。} 三个沙盒服务的静态语料规模为:

comment thread;Kiwix(\texttt{:8090})挂载 49\,GB 的
\texttt{wikipedia\_en\_all\_nopic.zim}(约 620 万英文条目,不含图片)

切换后端的 Agent 框架(GPT-Researcher、DeerFlow、Open-Deep-Research、
smolagents 等)都可在 0 行代码改动下接入。

\textbf{打分。}
我们对每份 Agent 报告(Markdown 长文)运行 7 个相互正交的 verifier,
给出 0--1 的 pillar 分数,再按 v3.1 权重做线性组合得到 composite:
\begin{align*}
  \mathrm{composite} =\ & 0.25\,\text{cite} + 0.20\,\text{evidence\_density}
    + 0.20\,\text{llm\_judge} \\
    & + 0.20\,\text{checklist} + 0.05\,\text{fact\_kg}
    + 0.05\,\text{structure} + 0.05\,\text{efficiency}
\end{align*}
每个 pillar 的具体做法:

\begin{itemize}[leftmargin=1.5em,itemsep=0.15em,topsep=0.2em]
  \item \textbf{\texttt{markdown\_structure}}\ (0.05):规范性检查。依据
    任务的 \texttt{markdown\_spec}(最小词数、最少段落数、最少引用数、
    最少浏览页数)做四项硬门槛判定,命中返回 1/4 的分数,线性加总。不涉及
    LLM,纯正则/计数。实现在 \texttt{src/verifiers/markdown\_report\_verifier.py}。
  \item \textbf{\texttt{citation}}\ (0.25):ALCE 风格的引用支持度。
    把答案切分到句子级,对每个含引用的句子,抓取其所挂 URL 的沙盒页面内容
    (通过 shim 的 \texttt{/scrape} 接口),判断句子的事实主张是否在被引页面
    中出现;以句子级 precision 汇总。默认走字符串覆盖率(prose mode);可
    切换到 NLI 蕴含模式(\texttt{CITATION\_MODE=entailment})。幻觉的 URL
    直接返回 0。
  \item \textbf{\texttt{fact\_kg}}\ (0.05):沙盒 ground-truth 事实召回。
    每道任务配套一份 golden 知识图谱三元组(由脚本离线从 Magento / Postmill
    数据库抽取出的\texttt{(product\_id, price, X)} 等),我们从 Agent 答案中
    抽取同模式的三元组(混合正则 + LLM 抽取),计算与 golden 的三元组
    recall / precision,取 F1。确定性、零方差,但受限于沙盒任务的 schema
    覆盖范围(scholarly 任务没有 shop/reddit 的三元组,返回 0)。
  \item \textbf{\texttt{llm\_judge}}\ (0.20):DeepResearch Bench 的 RACE
    四维打分 \citep{dr-bench-2026}——指令跟随度(instruction following)、
    内容深度(depth)、内容广度(breadth)、报告连贯性(consistency)。
    每维 0--10 整数分,均值归一到 0--1。Judge 模型按跨家族原则选:
    GLM-家族 Agent 由 DeepSeek-chat 判,DeepSeek-家族 Agent 由 GLM-4.5 判,
    从而抑制自偏好 \citep{wataoka-2024}。单次调用最大 2000 tokens,
    温度 0.2。本轮学术扩展的 24 次 rescore 统一用 GLM-5(中转 OpenAI-
    compat)。
  \item \textbf{\texttt{checklist}}\ (0.20):DRACO 风格的 15 项二值 rubric
    \citep{draco-2026}。rubric 条目按任务类别预置,例如 “报告是否恰好列出
    $k$ 个不同的产品”、“是否至少引用 $m$ 条不同的 reddit 帖子”、“是否显式
    标注价格区间”。每条由 Judge LLM 以 yes/no 回答,全答对为 15/15。实现在
    \texttt{src/verifiers/checklist\_verifier.py}。
  \item \textbf{\texttt{evidence\_density}}\ (0.20):单位词数内出现的
    \emph{独立}引用(按 URL 域名 + path 去重)数量。抑制 “一句话话 + 一个
    引用” 的灌水式引用,鼓励多角度支撑。低于 0.8 条/百词视为偏稀疏,高于 3
    条/百词视为过度引用(罚分)。
  \item \textbf{\texttt{efficiency}}\ (0.05):基于 tokens / 墙钟时间 / 估算
    成本三项,按任务预算(默认 300 秒 / \$0.10)线性归一。token 口径从
    Agent 的 \texttt{EfficiencyMetrics} 收集,兼容 Anthropic / OpenAI 的
    usage 字段。超过预算按比例扣分。
\end{itemize}

\textbf{composite 汇聚与排序。} 在 pillar 层面每 $(agent, task)$ 得到一个
composite;在 agent 层面,我们把任一两 agent 在同一 task 上的 composite
差值作为一次合成 “battle” 的胜负输入,用 Bradley--Terry 最大似然拟合得到
Elo 评分,1000 次 bootstrap 给出 95\% 置信区间。相邻排名的 permutation
$p$ 值也一并报告。此外,\texttt{llm\_judge} 还可以作为一个独立的 pair-wise
battle 信号,用 real-battle Elo 与 composite Elo 对齐以做交叉验证。

\section{扩展实验与基础设施}
\label{sec:infra}
%================================================================

除了主体的 14 $\times$ 8 评分矩阵外,DR-Arena 还配套了若干常被基准审稿
提及的扩展:

\paragraph{对抗语料(Adversarial Corpus)。}
为了检测 Agent 对\emph{不存在实体} 的行为模式,我们在
\texttt{data/tasks/deep\_research/adversarial/} 下加入了 5 道对抗任务
(\texttt{dr\_adversarial\_0001}--\texttt{0005}),每道任务在
\texttt{intent} 中要求 Agent 检索一个虚构的产品 / 论文 / Reddit 用户
(例如 \emph{``Acme Thunder-9 ANC 耳机''}),而这些实体在沙盒语料中并不
存在。配套的 \texttt{AdversarialRefusalVerifier}(\texttt{src/verifiers/
adversarial\_verifier.py})在答案中按三档打分:
\emph{拒答或显式承认找不到} = 1,\emph{含糊其辞或说 ``可能找不到''} = 0.5,
\emph{编造链接/参数/评论} = 0。该子集目前只跑了基础 smoke;全矩阵的
对抗行为测评留作后续工作。

\paragraph{WebArena factoid 适配(188 道)。}
\texttt{scripts/adapt\_webarena\_tasks.py} 把 WebArena 官方
\texttt{tasks.raw\_v2.json} 中 188 道 shopping-domain factoid 任务
转换到本基准的 schema,并附带一个
\texttt{src/verifiers/factoid\_verifier.py}(\texttt{string\_match}
判定器)。目的是在不依赖 LLM Judge 的情况下,用\emph{确定性的} string
级别准确率与主 rubric 做交叉验证。这 188 道任务目前仅作基础设施储备,
尚未纳入 14 个 Agent 的全矩阵跑。

\paragraph{Per-agent token budget guard。}
\texttt{src/budget/token\_guard.py} 给每个 run 强制加了
\emph{上限 200\,k input tokens / 20\,k output tokens} 的硬限,一旦命中
就提前终止并记录 \texttt{budget\_exceeded}。这一兜底把 GPT-5-chat-latest
的单 run 成本控制在约 \$0.04 以内,也是本轮观察到 reasoning-token
偷烧(见附录)后加入的防御。

\paragraph{Per-pillar Elo 与 real-battle Elo。}
除 composite Elo 外,我们还为每个打分维度(7 个 pillar + \texttt{llm\_judge}
独立 pair-wise battle)单独拟合 Bradley--Terry Elo。这使得在 composite 上
相邻的两个 Agent 可以从 pillar 层级比较优劣——例如 \texttt{camel-ai-ds}
的 \texttt{cite} pillar Elo 为 1141、\texttt{llm\_judge} pillar Elo 只有
770,两者差 370,揭示它的高 citation 密度并没换来 Judge 认可的报告质量。
完整 7 + 1 维 Elo 表见 \texttt{FINAL\_LEADERBOARD.md}。

%================================================================
\section{F1:LLM Judge 并不奖励篇幅}
\label{sec:length}
%================================================================

\paragraph{动机。}
长度偏好假设 \citep{wataoka-2024,arena-hard-2024} 认为,将 LLM 用作 Judge
会倾向于给更长、更自信的回答打高分。若干基准论文因此默认此偏差存在,要么做
长度归一化,要么只对长度匹配的 pair 汇报。

\paragraph{方法。}
对 80 次 run 的每份 \texttt{answer.md},我们统计其词数(先剔除 inline
代码与 URL),然后计算词数与每个打分维度(以及 composite)之间的 Spearman
秩相关。相关系数在全部 run 上联合估计。词数跨度 191--5384,均值 1540,存在
明显的 Agent 风格聚类(DeerFlow 系列均值 3162 词,smolagents 均值 991 词)。

所有基于 LLM Judge 的基准都把 Spearman 长度相关表作为一项常规健壮性检查
对外公开。

\section{F2:URL 幻觉是架构问题,不是模型问题}
\label{sec:halluc}
%================================================================

\paragraph{动机。}
一种流行叙事认为 “不确定时 LLM 会编造链接”。我们验证
URL 幻觉究竟是(a)在所有 run 上按比例分布的随机噪声,还是(b)集中在
少数 Agent 的架构性瑕疵。

\paragraph{方法。}
我们对 70 份 \texttt{answer.md} 做纯启发式扫描
(\texttt{scripts/error\_taxonomy.py}),把每一份划入四个互斥主类中的一个:
\begin{itemize}[leftmargin=2em,itemsep=2pt,labelsep=0.5em]
  \item \texttt{hallucination\_url}:引用 $\geq 2$ 个非沙盒域名的 URL,
    或包含五类 “虚假信号”正则(\texttt{(placeholder)}、
    \texttt{example.com}、公网 \texttt{wikipedia.org}、\texttt{*.tavily.*}、
    \texttt{onestopmarket.com}),或写出一份 $\geq 200$ 词却零 inline 引用的报告。
  \item \texttt{format\_error}:答案被 JSON 包裹但不足 400 词。
  \item \texttt{timeout\_or\_empty}:不足 50 词,或含字符串
    “failed to finish”。
  \item \texttt{ok}:以上都不符合。
\end{itemize}
该分类完全由正则匹配完成,不调用 LLM。判定标准偏严:软幻觉
(URL 真但正文事实错)会被漏判,硬幻觉(引用一个不存在于沙盒里的 URL)则
会被精准捕捉。

\paragraph{结果。}
\begin{table}[h]
  \centering
  \small
  \input{tables/halluc.tex}
  \caption{按主错误类型分层的 per-Agent 分布。四个 Agent(加粗)在全部
    4 次 run 上均触发 URL 幻觉;其余 9 个 Agent 在同一类上为 0/4。
    表格由 \texttt{data/results/error\_taxonomy.json} 自动生成。}
  \label{tab:halluc}
\end{table}

\paragraph{原因。}
表 \ref{tab:halluc} 同时否定了 “LLM 会幻觉” 与 “随机噪声”
两个解释:
\begin{itemize}[leftmargin=2em,itemsep=2pt,labelsep=0.5em]
  \item \textbf{同一 LLM、不同架构结果相反}。
    \texttt{camel-ai-ds}(DeepSeek)幻觉率 0/4;
    \texttt{gpt-researcher-ds}(DeepSeek)幻觉率 4/4。
  \item \textbf{同一架构、不同 LLM 结果相同}。
    \texttt{smolagents}(GLM-4.5)与 \texttt{smolagents-ds}(DeepSeek)
    都是 4/4 —— 此处架构中 \texttt{</code>} 与
    \texttt{<end\_code>} 的 sentinel 错配(现已修复,但本文汇报的 run
    在修复前)压倒了 LLM 选择的影响。
\end{itemize}
四个 Agent 各自的具体成因:
\begin{description}[leftmargin=2em,itemsep=2pt,labelsep=0.5em]
  \item[\texttt{gpt-researcher-ds}] RAG 组装阶段,当检索返回项数低于下限时,
    其模板会以 \texttt{(placeholder)} 填充 \texttt{\{sources\}},随后被
    LLM 当作真实 URL 写入正文。
  \item[\texttt{odr-ds}] summarizer 将来源 URL 视为 soft constraint,
    DeepSeek 会将其改写成形貌合理但实际不存在的链接。
  \item[\texttt{smolagents}(两种 LLM 后端均然)] 修复前的 sentinel 错配
    使得 Agent 尚未调用任何工具就停止,然后完全凭训练记忆生成正文并编造
    URL 以伪装出引用感。
\end{description}

\paragraph{启示。}
深度研究类 Agent 的 URL 幻觉并非 LLM 层面的属性,而是检索-组装 pipeline
的工程属性。对基准作者而言,应公开 per-Agent 的错误类型分布,而非只汇报
单一 composite;对 Agent 作者而言,修掉 URL 幻觉不需要换 LLM,改 pipeline
即可。

%================================================================
\section{F3:IRT 与 Elo 相互印证 —— 排名方法的自洽性}
\label{sec:irt}
%================================================================

\paragraph{动机。}
本节用一个独立的统计框架对 Elo 排名做交叉验证 —— 两参数逻辑斯蒂项目反应
理论模型(2-PL IRT,\citep{lord-1980})—— 以判断 Elo 排名是否只是 Elo
算法本身的 artefact。

\paragraph{方法。}
将 $14 \times 8 = 112$ 个 (agent, task) cell 中实际有分的 80 个 composite
分数按每道任务的中位数二分化:$Y_{ij} = 1$ 当且仅当 Agent $i$ 在任务 $j$
上的 composite 不低于任务 $j$ 在所有参与 Agent 上的中位数。拟合模型:
\begin{equation}
  P(Y_{ij} = 1 \mid \theta_i, a_j, b_j) =
  \sigma\big( a_j (\theta_i - b_j) \big),
  \label{eq:2pl}
\end{equation}
$\boldsymbol\theta$ 附加一个均值为零、方差为一的软罚项以保证可辨识。
2-PL 的似然在 $\boldsymbol\theta$ 与 $\mathbf{a}$ 同时变号下不变,所以
当 $\mathrm{corr}(\boldsymbol\theta, \bar Y_{i\cdot}) < 0$ 时我们翻转
两者符号,以与 “分数越高越好”对齐。另一个独立流程计算
Bradley--Terry Elo \citep{bradley-terry-1952}:对所有任务内的
$\binom{13}{2}=78$ 个 composite 成对胜负做 MLE。

\paragraph{结果。}
\begin{table}[h]
  \centering
  \small
  \input{tables/irt.tex}
  \caption{Bradley--Terry Elo 与 IRT 2-PL 下的 Agent 能力估计。
    两种方法下的排序 Spearman $\rho = 1.00$($p < 10^{-3}$)。
    题目参数(表中未列):四道任务的 $a_j$ 全部抵到区分度上限 5,
    且 $b_j$ 聚集在 $\approx -0.33$。
    表格由 \texttt{data/results/arena\_elo\_ED.json} 与
    \texttt{data/results/irt\_calibration.json} 自动生成。}
  \label{tab:irt}
\end{table}

\paragraph{原因。}
两种方法没有共享任何代码路径:Elo 是从成对胜负做顺序更新的累积方法,
IRT 是在潜在能力模型下做全局 MLE。它们给出相同排序,是对 “数据中
真的存在一个一致的排序信号” 的弱但独立的证据。

题目参数信息量更大。四道任务 $a_j$ 都顶到我们的 clip 上限,意味着这 4 道
题每一道几乎都能独立地完整还原 13 个 Agent 的相对顺序;它们的 $b_j$ 也都
挤在 $-0.33$ 左右,即难度相当。这提示的是 \emph{任务多样性不足}:当前
4 道题在同一条能力维度上测量得很好,但难以区分比如 “擅长长篇综合”
与 “擅长短答复查”两类 Agent。我们正在引入的学术扩展集($n = 20$,
覆盖医学、经济、哲学等不同领域)就是为了补上这一差异维度。

\paragraph{启示。}
目前排名并非 Elo 单一方法的 artefact;但 4 道任务的消费者子集信号虽强,
多样性不足,在写出 IRT 的定量 $a, b$ 参数声明之前需要更长、$b$ 分布更分散
的子测试。因此本文中 IRT 参数仅以 “方向性结论”(directional)
呈现,待 14 Agent $\times$ 24 题的矩阵就绪后再做定量声明。

\section{F4:决定源多样性的是架构,不是检索层}
\label{sec:sources}
%================================================================

\paragraph{动机。}
一个常见的简化假设是:给定相同的检索基础设施,两个 Agent 看到的就是相同
的语料,只在下游推理上有所不同。我们在同一 shim 下固定检索侧,直接测量
最终报告里究竟出现了哪些 URL。

\paragraph{方法。}
四道学术任务(\texttt{dr\_cross\_v3\_0088} 感冒医学、
\texttt{0095} 科学方法论、\texttt{0100} 通胀、
\texttt{0105} 诱导需求),每一道都要求在 Wikipedia 文章与
论坛讨论之间做交叉引用。shim 此时已接入 Kiwix(\texttt{localhost:8090})
与 Magento、Postmill 并列;一个典型的 \texttt{POST /search} 请求
(比如 “common cold rhinovirus virology”)返回 6 条结果,其中 3 条
是 \texttt{localhost:8090} 的 Wikipedia 条目,另外 3 条来自 shopping 或
论坛。

两个 Agent 共用同一个 shim URL 和同一套环境,唯一差异在其内部组装方式:
\begin{description}[leftmargin=2em,itemsep=2pt,labelsep=0.5em]
  \item[\texttt{gpt5chat}] OpenAI Agents SDK 风 tool-calling 循环
    (\texttt{scripts/run\_gpt5.py}):LLM 自己读一条整体搜索返回、筛选出
    有前途的 URL,再单独调用 \texttt{scrape(url)}。
  \item[\texttt{gpt-researcher}] 经典 RAG 组装:检索结果先经过一个子 LLM
    做打分与 rerank,再压缩入上下文喂给 writer LLM;writer 生成最终报告,
    并按 rerank 存下来的源做引用。
\end{description}

与 \texttt{citation} 维度(引用\emph{数量})正交。

\section{讨论}
%================================================================

四个发现的成熟度并不相同:
\begin{itemize}[leftmargin=2em,itemsep=2pt,labelsep=0.5em]
  \item \textbf{F1(长度)}:80 runs 下 $\rho = -0.289$,完整结论,可以用
    表 \ref{tab:length} 独立成段作为健壮性检查。
  \item \textbf{F2(幻觉分类)}:在 14 $\times$ 8 = 112 cell 中 34 条命中
    URL 幻觉,集中在 4 个 agent 上,结构稳定。下一步是将分类从正则升级到
    LLM 标注以覆盖软幻觉(改写、数字轻微篡改)。
  \item \textbf{F3(IRT)}:$14 \times 8 = 112$ cell 下拟合,但 4 道消费者
    任务全部顶到判别度 $a = 5.0$ 上限、4 道学术任务被标记 INVERTED 或
    LOW\_DISCRIM —— 这本身是一个\emph{结构性发现},而不是 IRT 失败:
    消费者与学术任务测不同的能力。正式定量 $a, b$ 置信区间需要更大的学术
    pool(目前 20 道中只取了 4 道)。
  \item \textbf{F4(源多样性)}:本轮 6 个 agent 在 scholarly 上的数据已经
    可以定量比较——\texttt{gpt5chat} 平均每篇 2.5 个 Wikipedia 条目,
    \texttt{gpt-researcher}(RAG 组装)0 个。完整 ablation 仍需把学术任务
    集扩大到与消费者对等的规模。
\end{itemize}

四个发现共享一个结构性质:它们之所以能够被观察到,完全依赖本基准采用
\textit{多维度分数 + 失败模式追踪}而非单一 success/failure 指标。
单分数的 live-web 基准无法触及它们中的任何一个:F1 需要 per-pillar
相关性、F2 需要在已知沙盒上的 URL 离线对照、F3 需要稳定冻结的语料以
构建全矩阵、F4 需要每份报告的完整 URL 出处。

\paragraph{评分系统的已知偏置(react 为什么排第一)。}
本轮 composite Elo 表里 \texttt{react-qwen35plus}(1313.6)与
\texttt{react-glm5}(1277.5)排名第一、第二,同底座的
\texttt{gpt-researcher}(GLM-4.5,1242.7)第三,两类差距约 70 分。细拆到
pillar 层(仅消费者 4 任务,排除学术任务对 mature 方案的拖累):
react-qwen35plus 的 \texttt{fact\_kg} 均值为 0.12、\texttt{checklist}
为 0.67;同底座的 \texttt{gpt-researcher} 分别只有 0.05 与 0.30。差距主要
来自这两个维度,而非 \texttt{markdown\_structure} 或 \texttt{llm\_judge}。

\textbf{这不是 react 的架构更好,是信息通道不对等。}
\texttt{glm\_react\_agent} 使用 Playwright 驱动真浏览器,并直接 import 沙盒
的 \texttt{envs.shopping.oracle\_dr.magento\_scrape} / \texttt{envs.reddit.
scrape} 这类\emph{本项目私有的 DB 抓取器},一次调用就能拿到
\texttt{\{product\_id, price, rating, description\}} 级别的结构化 JSON。
而 \texttt{gpt-researcher} / \texttt{deerflow} / \texttt{camel-ai} /
\texttt{smolagents} 只能通过 \texttt{search\_shim} 的 \texttt{/search}
(Tavily-兼容,返回的是预抓取的文本片段)和 \texttt{/scrape}
(Firecrawl-兼容,返回 markdown 全文)。后者把 \$299.99 变成 ``约 \$300''、
把 4.5 星变成 ``好评''。结果就是 react 的 answer 里留下了精确数字,
可被 \texttt{fact\_kg} 的 substring recall 和 \texttt{checklist} 的
``是否恰好列出 $k$ 个不同产品且各带评分'' 等 rubric 条目正中打中;
mature 方案只能写进近似描述,同一个 verifier 下分数自然低。

换句话说,verifier 本身是公平的(对所有 agent 跑同样的 LLM rubric / 同样
的三元组匹配逻辑),但 agent 看到的\emph{输入} 不公平。因此\textbf{排名
第一本身是一条关于 benchmark 的诊断信息,而不是关于 agent 的结论:}
当前 \arena{} 在奖励 ``为该沙盒定制的工具调用''。

\textbf{Ablation 量化。} 为了定量验证上述主张,我们做了一次消融:把
\texttt{fact\_kg} 与 \texttt{checklist} 两个 pillar 权重全部置 0,其余
pillar 按比例重归一,用同一套合成 battles 逻辑重新拟合 BT-Elo。排名变化
在表 \ref{tab:ablation} 中:\texttt{react-qwen35plus} 保持第一(1232.9),
但 \texttt{deerflow-glm46-shim} 从第 4 升到第 2、\texttt{deerflow-glm46-new}
从第 6 升到第 3、\texttt{react-glm5} 由第 2 降到第 4。也就是说,第 1--4
名的互换完全由 \texttt{fact\_kg + checklist} 两维驱动。

\paragraph{已落地 / 后续修法。}
\textbf{(1) 已实施}:给 \texttt{search\_shim} 增加了两个结构化 lookup
端点 —— \texttt{POST /post\_lookup}(返回 Postmill 帖子的

\texttt{envs.*.scrape} 等价的结构化数据,不再依赖项目 import 路径;已
随 Phase 10d commit 一起上线。
\textbf{(2) 待办}:在 \texttt{glm\_react\_agent} 中移除对
\texttt{envs.*.scrape} 的直连 import,改走 shim 的上述两个端点,把
\emph{信息通道} 强制对齐,剩下的差异才真正反映 agent 架构能力。
\textbf{(3) 待办}:报告 \emph{两张 leaderboard}:\textbf{Playwright 类}
(带真浏览器、可注入任意 JS/DOM 操作)与 \textbf{HTTP-tool 类}(仅走
shim 的 HTTP 接口),分别计算 Elo。这样读者能看到 ``同访问类''下谁更强,
避免把工具集差异误算成能力差异。

\subsection{踩过的坑(精选)}

\begin{enumerate}[leftmargin=2em, labelsep=0.5em]
\item \textbf{Clash fake-IP 劫持 SSH}(198.18.0.9 / 198.18.0.30):
  \texttt{*.vicp.fun} 在 TUN 模式下 TCP 通但握手被断,只能重试或走 ProxyCommand 直连。
\item \textbf{GPT-5.4 在 tool-loop 里偷烧 reasoning tokens}:
  单 prompt 看 reasoning=0 像 pure chat,多轮 tool-call 下突然开启,
  每轮烧 $\sim$750 output tokens。agent benchmark 必须用 \texttt{gpt-5-chat-latest}。
\item \textbf{GLM JSON mode 兼容性}:ODR / dzhng / smolagents 走 GLM OpenAI-compat
  时 \texttt{with\_structured\_output} 与 \texttt{generate-object}
  双双 ValidationError,催生 Phase 9 角色互换(agent 切 DeepSeek,judge 回 GLM)。
\item \textbf{DeerFlow custom adapter 截断 5000 字符}:原以为 DeerFlow 弱,
  走 shim 后 +162 Elo。教训:评测不能对框架降级。
\item \textbf{smolagents 哨兵格式错}:GLM 输出 \texttt{</code>} 代替
  \texttt{<end\_code>} 导致 code parser 每步失败,零 tool call 纯幻觉。
  修法:\texttt{CodeAgent(..., stop\_sequences=["</code>",\allowbreak{}"<end\_code>"])}。
\end{enumerate}
