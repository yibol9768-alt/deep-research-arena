基于沙盒环境的开源深度研究智能体评测基准

Deep-Research Arena · 答辩讲解稿 · 2026年5月

一句话:用三个本地化的密封服务 (Magento 电商、Postmill 论坛、Kiwix 离线维基) 替代开放互联网,把 13 个开源深度研究框架放到同一基底、同一骨干、同一打分栈上,以「URL 可达性作乘性门」的复合指标 composite_v2 排出 Elo 表,得到一项可被任何人在普通硬件上字节级复现的结论。

三句话陈述

  1. 开放互联网不是评测 DR 智能体的好环境,因为页面会漂、URL 会 404、真实性几乎不可探测。我们用三个冻结的本地语料把这套环境锁住。
  2. 评分体系里我们做的最重要的事是把「URL 可达性」从加分项升格为乘性门:智能体捏造 URL,无论文笔多好,分数直接归零。
  3. 这一改动直接反转了排名。GitHub 25.7k stars 的 gpt-researcher 在加性公式下第 1,在乘性门下掉到第 5;有显式 URL 校验流程的 camel-ai 升到第 1。

评委可能直击的三个点 (先把答案放在脑子里)

  • 「为什么不在 GAIA / BrowseComp 上跑就行?」答:它们是开放网,真实性不可探测,实验不可复现。详见 §1.1。
  • 「为什么真实性门要乘性而不是加性?」答:加性会把「文笔好的捏造」奖励上去,乘性能强制归零。详见 §7。
  • 「9 个框架的 Elo 名次,统计上真的能区分吗?」答:头部三名一簇、尾部六名一簇,中间隔一条 362 Elo 的鸿沟 (p<0.001),其余相邻名次置换 p 都 >0.30。详见 §8 与 §10。

1. 为什么做这件事

深度研究 (Deep Research, DR) 智能体的工作流是:读一个研究问题、上网搜索证据、抓取页面、写一份带引文的报告。我们要评估的能力是多面的:智能体是否找到了正确的信息源,是否忠实引用,是否做了跨源综合,是否承认了自身的局限。直觉上,衡量这些能力的自然环境就是开放互联网。GAIA、BrowseComp、AssistantBench、BrowserGym 也都是开放网基准。

本论文的核心命题:对以「科学测量智能体能力」为目标的研究而言,开放互联网恰恰是错误的环境。

1.1 开放网的四个问题

① 漂移

一周前还能支撑某项结论的页面今天可能 404,昨晚可能被修改过,今早可能被搜索引擎重排。两位研究者隔一个月跑「同一个」基准,得到的数字会不同,且双方都无法证伪对方。基准本身因此不可证伪。

② 真实性几乎不可探测

真实性打分要回答三个子问题:URL 能否解析、页面是否含被引用文本、被引页面是否真的支撑论断。在开放网上,这三个探针每一次都是与地理路由、登录墙、限流、CDN 抖动、静默改版的赌博。从智能体所在主机返回 200 的 URL,从打分主机看可能就是 403。

③ 产品评测 ≠ 能力测量

gpt-researcher (GitHub 25.7k) 在开放网基准上拿高分,有两种读法:架构好,或者它对开放网调得极顺 (提示词针对真 Google 调过、检索流水线针对真 CDN 工程化)。两者都是合理意义上的进步,但我们声称要测的只是前者。沙盒可以主动屏蔽掉后者。

④ 昂贵

开放网每次运行都要拉真实页面、撞真实限流、烧真实 token,而很多页面下月就不在了。沙盒可以在 15 分钟内复跑一个智能体在一个任务上的运行、确定性打分、随时复现。

这场交易的本质是 用「真实感」换「可复现性」。沙盒覆盖面不如开放网 (本文三个语料共约 2000 个商品页 + 若干万 Reddit 评论 + 若干 Wikipedia 文章),但被牺牲的覆盖面换来了:每一个真实性探针都可以在 O(1) 时间内对冻结的真相执行,且 404 与网络抖动之间不再需要主观判断。

2. 整套系统怎么搭的

2.1 端到端数据流

整个基准由 三个沙盒服务 + 两个粘合服务 + 一份任务语料 + 一套打分栈 + 每框架一份运行器 构成。所有组件运行于一台 Windows + WSL 主机 (内部代号 westd),全部通过 localhost 通讯。

智能体进程只能看见 localhost。除搜索 shim 与 LLM 代理外,网络出口全部被屏蔽。每一次搜索调用都从智能体的虚拟环境出口到 8081 端口;每一次模型调用都出口到 8088 端口。智能体产出的每一个 URL 都会与三个沙盒 host:port 对作严格比对 (localhost:7770、localhost:9999、localhost:8090);任何指向其它地址的 URL 都不计入引文覆盖与可达性。这一条规则,是真实性可被打分的根本原因。

3. 三个沙盒服务

沙盒围绕三个属性而设计:智能体可能引用的每一个 URL 都可确定性地解析,三个语料彼此差异足够大以使跨源任务非平凡,整套栈在单机上不需要任何实时互联网调用。每个语料都是真实 DR 信息基底的「最小可行片段」。

3.1 Magento (shopping, localhost:7770)

电商语料是从 WebArena 的 onestopmarket 快照导入的 Magento 2 实例,约 2000 商品页,横跨 Electronics、Home & Kitchen、Beauty & Personal Care 三大类目和数十个叶子类目。每页含 URL slug、价格、星级、评论数、自由文本特征列表、若干用户评论。

为什么选 Magento? 它是 WebArena 用的真实电商基底;商品页 URL 结构稳定地以 /product/...html 结尾;直接复用了 WebArena 的「类目页 → 商品页」浏览模式。

两个实践后果。 ① Magento URL 路径大小写敏感(默认设置)。把路径转小写的框架自动失去可达性, ldr 的失败模式之一就是这条。② 并行 HEAD 探活时 Magento 偶尔重置连接。URLReachabilityVerifier._probe 因此采用 GET (非 HEAD) + 3 次重试 + 指数退避。

3.2 Postmill (reddit, localhost:9999)

论坛语料来自 WebArena 的 Postmill 实例,含一批精选「子版」(/f/technology、/f/gadgets、/f/AskReddit、/f/headphones、/f/askscience 等),含帖子、评论、点赞数、时间戳。

为什么选 Postmill? WebArena 自带的开源 Reddit 克隆;URL 结构 /f/<forum>/comments/<post_id>/<slug> 稳定;评论层级让智能体能从静态商品页拿不到的「社区情绪」中获取信号。任务中的「按 Reddit 情绪给品牌排名」「评分 vs 情绪分歧」等综合要求都依赖这个语料。

3.3 Kiwix (Wikipedia, localhost:8090)

百科语料是 Kiwix 服务器加载 wikipedia_en_all_nopic ZIM 文件,文章 URL 形如 localhost:8090/content/wikipedia_en_all_nopic/A/<Article_Title>。

为什么选 Kiwix? ZIM 文件是英文维基的冻结签名快照;Kiwix 服务器轻量且确定;维基镜像是用以核验技术性论断 (编解码器、电池、RFC) 的天然「百科基底」。

两个实践后果。 ① 文章路径大小写敏感:/A/Bluetooth 可达,/A/bluetooth 返回 404。若干框架默认归一化为小写,在维基域上单独拿到 reachability=0;ldr 排名靠后部分原因即此 (其 30 份打分 composite_v2 均值 = 0.029)。② 部分框架会输出 en.wikipedia.org/wiki/<Title> 形态。HTTP 层拦截 (scripts/run_deep_task.py:107-115) 把这些在传输层重写为 Kiwix 前缀;若无此重写,langchain-odr 与 ii-researcher 会输出充满外部 URL 的报告。

3.4 搜索 shim (localhost:8081)

Tavily 兼容的 HTTP 服务,实现于 integrations/search_shim/app.py。对外暴露 POST /search 与 POST /extract 两个端点。内部把查询路由到三个索引 (商品标题、论坛主题+正文、维基标题+导言段落),合并 top-N,以 Tavily JSON 返回。

三条重要性质:

  • Tavily 兼容即集成契约。 接入的每一个框架都已经支持 Tavily。把每个框架的 Tavily 客户端指向 shim 是一行 base_url 配置即可完成的事,无需写自定义工具或调提示词。
  • 无任何真实互联网调用。 shim 从不发出对外网的网络调用。智能体搜索沙盒不存在的品牌就是空列表。
  • 排序稳定。 按固定 BM25 排序,第 0 分钟与第 1000 分钟的同一查询返回完全一样的结果。这一性质是每任务黄金池得以作为冻结真相的基础。

3.5 DS proxy (localhost:8088)

OpenAI 兼容的大模型端点,实现于 integrations/ds_proxy/app.py。智能体把它当作 https://api.openai.com/v1 用,代理把每个请求翻译为 DeepSeek API 调用,在请求体中注入 thinking: disabled,然后流式回传。

集中化 LLM 的三条理由:

  • 全基准一个骨干。 头部排名中每个智能体都跑在同一份 deepseek-v4-flash 非推理 checkpoint 上。跨智能体对比因此测量的是智能体架构,而非骨干强度。
  • 服务端持有密钥。 智能体只看见 OPENAI_API_KEY=anything。真实 DeepSeek 密钥存于代理的环境变量中,永不进入任何智能体虚拟环境。
  • 单点关闭推理。 部分框架 (langchain-odr、qx-agents) 会因为模型名子串而静默切换到推理模式。代理层注入 thinking: disabled 拦截每一个请求,保证比较是在同一份非推理 checkpoint 上。

代理同时充当判官端点。judge_client.py 读 JUDGE_BASE_URL=http://localhost:8088/v1,每次 LLM 判官调用 (checklist、citation_alignment、presentation、analysis_depth、claim_nli) 都打到同一份代理、同一份骨干、temperature: 0。

4. 任务怎么设计

任务语料是 100 个跨站点深度研究任务,其中 57 个完成端到端打分。每个任务是 data/tasks/deep_research/cross_site_deep/ 下的一份 JSON,内含:

  • 自然语言意图 (通常 30 句左右)
  • 引文政策:每域最小、总量最小、必须域名列表
  • Markdown 规范:字数 / 引文数 / 段落数
  • URL 覆盖设置:黄金池路径、recall / coverage 阈值
  • URL 可达阈值
  • 黄金三元组文件路径
  • 可选的综合要求
  • 每任务 21 项 checklist 的路径

三条任务设计常量

  1. 每任务三个站点。 每个跨站点任务都要求引用全部三个沙盒语料中的页面。citation_policy.per_domain_minimum 块规定每个域最少多少不同 URL (耳机任务的代表性切分为 shopping 30、reddit 20、wikipedia 15)。
  2. 引文总下限 120。 每任务在 Markdown 内至少 60 个引文,黄金池中的 must_cite_urls 至少 120 条。120 这一数字是「可见性下限」:它强制智能体真正去做跨源检索并寻找独立证据,而非吐出 Tavily 的前三条结果。
  3. 冻结的黄金池。 每个任务一份 data/golden/deep/dr_cross_deep_XXXX.json,由 scripts/build_deep_golden.py 一次性从干净沙盒快照中爬取。文件含 must_cite_urls (智能体应当命中的 URL,带权重)、expected_pool_urls (更广泛语料)、triples (主-谓-宾-source_url-quoted_span 五元组,驱动 claim_nli)。两份池一次性爬取,从不重生成,recall 分母被冻结。

代表性任务: dr_cross_deep_0001 (消费级耳机市场情报)

意图 31 句,4144 字符,要求:

  • A 节 (产品图景): ≥40 个不同商品页,跨 ≥6 个品牌、3 个价位区间;每件给出价格 / 星级 / 评论数 / 特征论断。
  • B 节 (社区情绪): ≥30 个 Postmill 帖,跨 ≥4 个子版,分为 5 类情绪 (praise / complaint / technical_question / comparison / purchase_advice)。
  • C 节 (技术依据): ≥25 篇维基文章支撑 A 节技术论断;9 个指定文章 (Active noise control、Noise-cancelling headphones、Bluetooth、AptX、LDAC、Headphones、Loudspeaker、Lithium-ion battery、Wireless microphone) 强制覆盖。
  • D 节 (跨源综合): ≥5 处商品特征与维基矛盾;按 Reddit 情绪给品牌排名;≥3 处「评分 vs 情绪」分歧;一份 top-10 购买建议,每条 ≥1 shopping + 2 reddit + 1 wiki URL。

该任务的黄金池为 must=125, pool=739。它是三个「推荐锚点」之一 (0001 / 0002 / 0005),用于测试智能体能否给出排序型交付物;其余任务覆盖比较、辟谣、因果分析、时间线、枚举等模式。

每任务 21 项 checklist,均人工撰写,集中存于 data/tasks/deep_research/cross_site_deep/checklists_deep.json。

5. 13 个框架的接入

5.1 三种适配范式

每个框架最终都跑在同一份 shim 与同一份 ds_proxy 之上。适配层分为三类:

进程内

in-process

在 scripts/run_deep_task.py 中定义函数,直接 import 框架并在 .venv-camel 内运行。Tavily 客户端通过 monkey-patch 重定向。

camel-ai、smolagents、langchain-odr、flowsearcher-ds

子进程

subprocess

写一份 driver 脚本到 /tmp/,以另一份 Python (该框架的虚拟环境) 启动。适用于依赖冲突严重或必须使用其自身配置机制的框架。

gpt-researcher、ii-researcher、deerflow、ldr、qx-agents、tongyi、storm、deepagents

Clean runner

scripts/runners/

自包含模块,导出 async def run(intent, model, shim_url, proxy_url) -> str,由 registry.py 自动发现,合并到 RUNNERS 分派表。

新接入框架的推荐路径

6.1 url_coverage:智能体是否引用了它应当引用的 URL

这是三个不同的子分加权求和,默认权重 0.55 must-cite + 0.15 pool + 0.30 domain。三项各自抓不同的失败模式:

子分权重公式它在问什么
must_cite_recall 0.55 Σ w_u (cited ∩ must) / Σ w_u (must) 黄金池里被点名要求必引的那批 URL,智能体命中了多少 (带权重)。耳机任务里 Bluetooth、AptX、LDAC、Noise-cancelling headphones 这 9 篇维基就是 must_cite,不引等于丢分大头。
pool_coverage 0.15 |cited ∩ expected_pool| / |expected_pool| expected_pool 是「称职智能体应该能找到」的更广泛语料 (每任务通常 600+ URL,例如 0001 是 739)。pool 比 must_cite 大一个数量级,允许智能体找到同等质量的替代页面也能拿分,而非死板只奖励 must_cite 精确命中。
domain_balance 0.30 min_d min(1, cited_d / min_d), d ∈ {shop, reddit, wiki} 三个语料里最弱那一个的填充率,取 min,不是平均。任务要求 shopping≥30 + reddit≥20 + wiki≥15;你引了 100 个 shopping 但只引了 5 个 wiki,wiki 填充率 5/15 = 0.33,domain_balance 直接就是 0.33。这是跨源任务的真正抓手。

为什么三项必须并存

只用 must_cite_recall?

奖励「命中冻结的黄金池」,但惩罚找到同等质量的替代页面的智能体。黄金池是一次性爬取的,不可能枚举所有合格 URL,过度依赖会让基准变得僵化。

只用 pool_coverage?

奖励「广撒网」,但智能体可以全部从 shopping 域里堆 200 个 URL 把分数堆上去,测不出跨源能力。pool 默认不区分三个语料的分布。

只用 domain_balance?

每个域只引 1 个 URL 就能拿满 (1/min 截断到 1),测不出深度。但凡引文上限不够,深度任务就完不成。

三个一起 must_cite 测「精确命中关键证据」,pool 测「广度合格」,domain 测「跨源平衡」,任何一个失衡都会被对应子分抓出来。min(1, cited_d/min_d) 的截断保证「过度堆 shopping」不能补偿「wiki 缺席」。

Pillar 通过阈值 (四项都满足才算 url_coverage 通过)

must_cite_recall ≥ 0.45
pool_coverage ≥ 0.12
domain_balance ≥ 0.80
unique cited URL ≥ 60

URL 在比较前经 canonicalize_url 规范化 (剥尾斜杠、域名转小写、丢 fragment、query 排序),避免 http://localhost:7770/foo/ 与 http://localhost:7770/foo 被算成两个 URL。源码在 src/verifiers/url_coverage_verifier.py:166-186。

6.2 url_reachability:抵抗 URL 捏造最强的单一信号

这一项在测什么 (直觉)。智能体在报告里写了一堆 localhost:7770/...html,但这些字符串究竟是真的指向一个能打开的页面,还是模型脑补出来的?这一项就回答这个问题:我们拿到智能体引的每一个 URL,亲自去 HTTP 请求一下,看返回 200 的比例是多少。

具体怎么做。验证器走五步:

  1. 从 Markdown 里把所有 URL 抽出来,既识别 [label](url) Markdown 链接,也识别裸 https://...。
  2. 过滤到沙盒域。判断条件是 host:port 严格匹配 三个沙盒端点,不是子串匹配。这一点很关键,如果用子串,localhost:7770 会匹配到 localhost:77703 这种攻击性输入,智能体可以蒙混过关。
  3. 每个 URL 经 canonicalize_url 规范化后查重,只对唯一 URL 打一次。
  4. 对每个唯一 URL 发起 GET 请求 (不是 HEAD,因为 Magento 偶尔在并行 HEAD 下重置连接),6 秒超时,3 次重试,指数退避 (0.5 s → 1 s → 2 s)。
  5. 分子记 200,分母记所有非 5xx、非 net-fail 的探针。
reachability = |{u : status(u) = 200}| / |{u : status(u) ∉ {5xx, net_fail}}|

为什么分母不是「总 URL 数」?因为我们要把智能体的罪和沙盒的罪分开。如果 Magento 在打分时挂了,所有探针返回 500,这不是智能体编造 URL,是基础设施事故,我们不能因此判智能体死刑。所以验证器额外维护两个标志:

  • infra_failure = True:所有探针都是 5xx 或网络失败 → 打分作废,这条 score 被退化过滤器剔除。
  • infra_warning = True:≥30% 探针是 5xx 或网络失败 → 仍打分,但标记给审计页。

关键拆分:4xx vs 5xx。4xx (尤其 404) 是「智能体编了一个不存在的 URL」,这是 reachability 应该惩罚的;5xx 是「URL 路径模式对,但服务器此刻挂了」,这不该惩罚。分母排除 5xx,意味着「智能体捏造 URL」(高 4xx) 与「沙盒挂了」(高 5xx) 走不同的处置路径,这是这一项设计上最重要的细节。

通过条件:reachability ≥ 0.30。

这一项在排名里起的作用。它是 composite_v2 的乘性门。reach = 0 ⇒ 整个 composite = 0,无论文笔多好、checklist 通过几项。真实性门反转排名 (§11.1) 这一头号发现,公式上完全来源于这一项。gpt-researcher 在任务 0009 上引 93 个 URL 只有不到 10 个返回 200,reach ≈ 0.10,达不到通过阈值,整个 composite_v2 归零。

源码:src/verifiers/url_reachability_verifier.py:55-82。

6.3 quote_match:URL 存在,但页面里真的有那句话吗?

这一项在测什么 (直觉)。reachability 只保证 URL 能打开,不保证页面真的支撑论断。我可以引一个能 200 返回的维基「蓝牙」条目,然后旁边瞎写一句「蓝牙规范由 IEEE 制定」(其实是 SIG)。URL 真,论断假,reachability 抓不到这种错误。quote_match 是设计上最便宜的支撑性检查:不调 LLM,不做 NLI,只看「论断的关键词在页面里出现了没」。

具体怎么做。三步:

  1. 从 Markdown 抽出每一个 (url, claim_context) 对。这里 claim_context 是引文位置前后 200 字符的窗口,把 Markdown 链接语法 [label](url) 替换为可见的 label 文本。
  2. 对每个唯一 URL 抓取页面正文,与 reachability 共用同一份带重试的 _fetch 助手。
  3. 对每个 (claim, page) 对,计算 token 重叠:把 claim 与 page 都小写化、按空白切词、做 stop-word 过滤 (去掉 the / of / and / is / a 这类),然后看:
overlap = |tokens(claim) ∩ tokens(page)| / |tokens(claim)| ≥ θ = 0.10

满足 overlap ≥ 0.10 算「这一对通过」,得分 = 通过对数 / 总对数。

为什么阈值定在 0.10 这么松?因为 200 字符的 claim 窗口里大多数是连接词和句法骨架,真正能与页面匹配上的是实体词与数字。要求超过 10% 的非停用词重叠,通常意味着至少 2-3 个实体或数字命中,这已经是非常显著的「页面里真有那个东西」的证据。再高就会假阴性 (页面用同义词表述同一事实),再低就漏 (任何带「Bluetooth」的论断都能蒙过去)。

为什么不直接做 NLI?NLI 是 §6.4 在做的事 (claim_nli),那一项很贵 (每对一次 LLM 调用,每报告耗时 30-120 秒) 且在我们的判官上不稳。quote_match 是 NLI 的便宜近似:精度低于 NLI 但召回高 (有页面文本支撑的论断,几乎一定能在词袋上重叠),最重要的是确定性 (跑两次给同样结果)。一个便宜的、确定性的、召回高的探针,在排名复合分里比一个昂贵且会假阴性的探针有用得多。

这一项在排名里起的作用。它进入 composite_v3 (权重 0.20),不进 composite_v2。但它在 per-pillar Elo 表里有独立一列,头部三框架的 quote Elo 都在 1300+,尾部六框架都在 800 上下,quote_match 与 reachability 的两根 Elo 轴是诊断「文笔好但捏造」的最清晰组合 (高 url_cov、低 reach、低 quote = gpt-researcher 的画像)。

源码:src/verifiers/quote_match_verifier.py:91-95。

6.4 claim_nli:重型 NLI 探针 头部已弃,但仍计算

这一项在测什么 (直觉)。quote_match 是「页面里有没有那些关键词」,claim_nli 是「页面那段话真的逻辑上支撑这个论断吗?」。这是 ALCE / RAGAS 文献里的标准支撑性探针,理论上比 quote_match 严得多。

具体怎么做。对每一个在黄金池里带 quoted_span 的 (claim_context, cited_url) 对 (注意:不是任意 URL,是黄金池已经为它准备了引文片段的那些),验证器经 ds_proxy 询问 DeepSeek V4-flash:

CLAIM: {claim_context}
SOURCE_QUOTE: {quoted_span from golden triple}
Does the source quote DIRECTLY support the claim?
Output JSON: {entail: bool, prob: float, reason: str}

零温度下,认为 prob ≥ 0.80 视为支撑 (与 ReClaim 论文阈值一致)。得分 = 支撑对数 / 总对数。每报告 max_calls = 80 上限以约束成本,耗时 30-120 秒/报告。

2026-04-27 起从头部 composite_v2 中剔除,乘数置 1.0。原因是 DeepSeek V4-flash 在视觉噪声大的商品页 (大量「Add to cart」「Customer review」「Related products」这类页面骨架文本) 上产生大量假阴性:页面明明含支撑文本,判官说不支撑。我们试过把页面正文先抽出来再喂判官,改善有限。最终的处置是:维度仍每次运行被计算并持久化进 score JSON,以便未来换更强 NLI 判官 (GPT-5、Claude) 时可以直接重打分而无需重跑智能体;但头部 composite_v2_truthful 的乘数置 1.0,不让它扭曲排名。

这一项在排名里起的作用。当前几乎为零 (头部弃用,v3 也不直接用)。它是一个未来的接口:换非 DeepSeek 的判官后,可以直接接回 v2 而无需重跑实验。

源码:src/verifiers/claim_nli_verifier.py;头部禁用见 src/scoring/leaderboard_composites.py:71-81 中 composite_v2_truthful 的实现。

6.5 checklist:21 项每任务专属的二值细则

这一项在测什么 (直觉)。前四项 (url_coverage、url_reachability、quote_match、claim_nli) 都是跨任务通用的真实性探针,它们不看任务到底要做什么。但每个任务有具体要求:耳机任务要求「按 Reddit 情绪给 ≥4 个品牌分类」「列出 ≥5 处商品与维基的矛盾」,这些是任务自带的强约束,通用探针测不出来。checklist 就是为每个任务手写一份 21 项的检查表,逐项判智能体报告通过没有。

21 项是怎么来的。每任务 21 项,人工撰写,集中存于 data/tasks/deep_research/cross_site_deep/checklists_deep.json。撰写时遵循三条原则:

  • 必须二值判定。每一项的措辞要让判官明确回答 PASS / FAIL,不能模糊。「报告读起来好不好?」这种问题不出现在 checklist 里 (它去 §6.7)。
  • 必须可在报告内验证。不能要求「智能体应该想到 X」这种心理状态,只能问报告里有没有 X。
  • 必须任务专属。「报告引用了 ≥3 个 Reddit 帖」属于通用 (它在 url_coverage 里);「报告分析了 Sennheiser 与 Bose 在主动降噪上的技术路线差异」才是 0001 任务专属的 checklist 项。

判官调用细节。经 ds_proxy 调零温度 DeepSeek V4-flash,system prompt 要求逐行输出 PASS / FAIL / UNCLEAR,UNCLEAR 算 FAIL。一次调用喂全部 21 项 + 完整报告,模型一次输出 21 行判决。

但 LLM 判官在长文本上不可靠 (这是文献里反复报告的问题:松懈模式、自我偏好、长度偏好)。我们用三层确定性守卫把判官包起来,凡是被守卫降级的项,不管判官说什么,统一记 FAIL:

#守卫触发条件抓的是什么 bug
1退化答案守卫 答案为空 或 ≤50 词且无引文 抓 (qx-agents error: ValidationError ...) 这种 placeholder。早期实验里,判官会对一个 30 词的错误消息高兴地报「21 项全 PASS」(因为它把错误消息当作「简洁的总结」)。这条直接跳过 LLM 调用,全 FAIL。
2全 PASS 短答案守卫 判官报 21/21 PASS 且 答案 <500 词 抓 DeepSeek V4-flash 在短输入上的「松懈模式」:输入太短,模型倾向于「都对吧应该都对」。500 词是经验阈值 (一份合格的 DR 报告至少应该 3000+ 词)。这条把全 PASS 降为全 FAIL。
3锚定关键词交叉检查 URL 可达性 < 0.30 且 项内含「URL / cited / linked / sandbox / domain / reddit / wikipedia / shopping / thread / article / page / forum」之一 抓「文笔好的捏造」。如果一个项问「报告是否引用了 ≥30 个 Reddit 帖?」,判官读了报告说 PASS,但实际 URL 都打不开,这一项就被守卫降级 FAIL。这条直接阻止了 gpt-researcher 在任务 0009 上拿 21/21 PASS:它引 93 个伪 URL,reach ≈ 0.10,所有带锚定关键词的 checklist 项全降级。

得分。pass_count / 21,最终落在 [0, 1]。

这一项在排名里起的作用。它进入三个复合分(v1、v2、v3),权重 0.40。是 Q = 0.40·url_cov + 0.40·checklist + 0.20·spec 中的两个 0.40 之一。意味着任务专属的完成度和跨任务的引文覆盖度权重等同,这是有意为之:URL 引对了但任务要求没做,跟任务做了但 URL 是瞎编的,在 quality 项上同样扣分;真实性门 (v2) 再额外乘上 reach。

源码:守卫在 scripts/score_deep_answer.py:111-180,checklist 数据在 data/tasks/deep_research/cross_site_deep/checklists_deep.json。

6.6 citation_alignment:ALCE 风格的每引文 NLI

这一项在测什么 (直觉)。checklist 看的是「报告整体完成度」,quote_match 看的是「引文位置周围 200 字符里有没有页面词重叠」。citation_alignment 比这两个都细:对报告里每一句带引文的话,我们拉取被引页面,问判官「这一整页是不是支撑这一句话?」。这是 ALCE 论文的标准范式 (per-citation precision / recall)。

具体怎么做。四步:

  1. 从 Markdown 抽 (sentence, cited_url) 对。一句话引了 3 个 URL,就拆成 3 对。同时识别两种引文形态:Markdown 链接 [label](url) 与裸 https://... (裸 URL 形态是 ldr 触发 quote_match 返回 claims_total=0 的真实原因,所以这里必须双形态支持)。
  2. 对每个唯一被引页面,用 4-worker 线程池抓取并剥 HTML:移除 script / style / nav / footer / header / noscript / aside / svg,只留正文文本,截断到 8000 字符 (足够覆盖典型 Magento 商品页与维基导言段,又能控住 LLM 上下文成本)。
  3. 对每一对询问判官 (零温度 DeepSeek V4-flash,max_tokens = 80): 
    PAGE TEXT: {first 8000 chars of cleaned page}
CLAIM: {the sentence}
Does the page SUPPORT the claim?
Output exactly: VERDICT: SUPPORTED  or  VERDICT: NOT_SUPPORTED
  4. 每对返回 SUPPORTED / NOT_SUPPORTED。

分数怎么算。验证器同时算 precision 和 recall,但头部得分取 precision:

precision = supported_pairs / total_pairs recall = sentences_with_≥1_supported_cite / sentences_with_any_cite

为什么取 precision 不是 recall?precision 抓的是「智能体引的页面真的支撑论断的比例」,这就是 ALCE 论文的「引文精度」;recall 是辅助信息,告诉我们「带引文的句子里有几句至少有一个有效引文」,在 deep-research 设定下 recall 几乎总是高 (因为每句都引了 2-3 个,只要一个对就 recall 加 1),信息量低。precision 是真正区分「胡乱引一堆 URL」与「精挑细选引对页面」的轴。

资源上限。每报告 200 对、每页 8000 字符、4 worker 线程池、max_tokens = 80。200 对的上限是约束典型 DR 报告 (60-150 引文 × 平均 1-2 句话各引一次) 的判官调用预算,超过 200 对则随机抽样 200 对。

这一项在排名里起的作用。它只进 composite_v3 (权重 0.15)。不进 v2 的理由跟 §6.4 类似:这是一个昂贵的 LLM 判官指标,放进头部排名会让结果对判官选择敏感。它的价值是审计:在审计页可以按 citation_alignment 排序,直接看到「哪些智能体的报告里引了对但引文与论断不对应」(例如,引了 Bluetooth 维基,但旁边的论断是关于 LDAC 编解码)。

源码:src/verifiers/citation_alignment_verifier.py:321 (取 precision 那一行)。

6.7 analysis_depth 与 presentation:报告作为研究交付物读起来如何

这两项在测什么 (直觉)。前面五项都在测引文质量,但一份 DR 报告还有另一面:它读起来像不像研究报告?有没有结构、有没有过渡、有没有跨源综合、有没有反驳意识、还是只是一长串「Page X says Y」的清单堆叠?这两项就测这个。注意:它们是「读得怎么样」,不是「真的吗」 ,所以不进头部 composite_v2,只进 composite_v3 (各权重 0.05 和 0.10)。

两者结构相同:Tier A 是确定性结构检查,Tier B 是 LLM 评分细则,然后按 0.3/0.7 (depth) 或 0.4/0.6 (presentation) 加权。Tier A 是「答辩底线」,Tier B 是「真正的水准」。

analysis_depth (10 项,4 + 6)

Tier A 确定性 (4 项):

  • 至少 1 个段落含 ≥2 个不同域的 URL (跨源段落存在)
  • 至少 3 个段落使用「矛盾 / contradict / conflict / disagree / discrepancy」类语言
  • 至少 3 个段落被 ≥3 个跨 ≥2 个域的 URL 支撑 (三角证据存在)
  • 比较性语言密度 (compare / versus / whereas / however) ≥ 0.15

域提取在 host 为 localhost 时使用 host:port,这样三个沙盒不会塌缩成一个域。正则在 analysis_depth_verifier.py:38-52。

Tier B LLM 判官 (6 项):超越枚举、因果推理、信息源三角、新见解、反驳意识、可行动结论。判官读完整份报告,逐项 PASS/FAIL。

得分: 0.3 · tier_a + 0.7 · tier_b。Tier B 权重高,因为它真正反映 DR 的核心能力 (综合、推理),Tier A 只是「至少别堆叠清单」的底线。

presentation (12 项,6 + 6)

Tier A 确定性 (6 项):

  • 标题层级 ≥3 (有 h1 / h2 / h3)
  • ≥5 个 h2 节
  • 节平衡 CV < 1.5 (没有一节占全文一半,其他节几乎空)
  • 至少一份列表或表格
  • 无 200 词以上的无标题孤立段 (大块未分节文本)
  • Flesch Reading Ease ≥ 30 (英文可读性下限)

Tier B LLM 判官 (6 项):逻辑流、过渡质量、无元叙述废话 (「In this section we will discuss...」)、格式一致、结论综合、专业语气。

得分: 0.4 · tier_a + 0.6 · tier_b。Tier A 权重稍高,因为结构性问题 (没标题、巨大孤立段) 是无可争辩的硬伤;Tier B 是更主观的「行文气质」。

长报告处理细节:Tier B 提示词对 >6000 词的报告送「头 + 尾」而非只送头,这样 conclusion_synthesizes 与 consistent_formatting 能真正看到报告结尾 (presentation_verifier.py:217-223)。

这两项在排名里起的作用。每次运行都被计算并写入 score JSON,但只对 composite_v3 有贡献。头部 composite_v2 主动不纳入它们。这是一个有意的设计决策:如果把「专业语气」「逻辑流」与「URL 可达性」放到同一根坐标轴上,真实性门反转的现象就看不到了,因为「文笔好的捏造」会被「文笔分」补偿回来。我们要分开这两件事,所以 v2 的乘性门只乘 reachability,不乘 presentation。v3 把它们都加上,提供一个「更柔和」的排名信号,让审计页可以排序「v2 上为 0 但报告非空」的智能体,使它们不静默消失。

6.8 七项怎么配合 (一张总览图)

维度抓的失败模式谁能蒙过这一项谁蒙不过
url_coverage「没引到该引的页面」引大量乱七八糟 URL 凑数真去检索每个域的关键证据

7. 七项怎么合成一个指标

到这一节为止,每份报告都已经在 7 个维度上分别拿到了一个 [0, 1] 之间的数 (见 §6)。但 Elo 对战只能在一对单一数字之间分胜负,我们因此还要做最后一步:把这 7 个数压缩成一个能够参与对战的复合分。这一节就完整讲这一步是怎么做的。

需要先说明的是,我们没有只生成一种压缩方式,而是同时计算了三种,叫做 composite_v1、composite_v2_truthful、composite_v3,全部写进每份 score JSON 里。原因是「该不该把可达性当门」「该不该把表达分纳入头部」都是方法学选择,任何一种合成都隐含了某种价值判断;把三种公式都计算并对外公开,让任何审查者都能用同一套底层 pillar 数自己重新合成一遍,确认结论不来自合成口径的暗中调整。头部排行榜实际驱动 Elo 的是 composite_v2_truthful,另外两个共存只为了对照与审计。

合成前的最后一步:把规范检查折成一个数 spec

开始合成之前,先把一个小数字定义清楚,它会在三个公式里都出现,叫 spec。spec 不是独立 pillar,它是三个最便宜的 Markdown 规范二值标志的算术平均,起到「报告至少有形可言」的底线作用:字数有没有达到任务规定的下限叫 words_ok,不同引文数有没有达到下限 (典型 ≥ 60) 叫 citations_ok,段落数有没有达到下限叫 paragraphs_ok,每个标志都是满足返回 1、不满足返回 0,最后

spec = (words_ok + citations_ok + paragraphs_ok) / 3

取值因此只能落在 {0, 0.33, 0.67, 1.00} 四档之一。这个粒度刻意做得粗,因为它的作用只是「过最低门」,不参与精细排名。源码:spec_pass_fraction,src/scoring/leaderboard_composites.py:28-35。

核心构件:质量项 Q

头部复合分的所有逻辑都围绕一个数展开,我们叫它「质量项」 Q。它是三个数的加权和,把「引文范围对不对」、「任务要求做没做」、「报告基本形态对不对」三件事压成一个 [0, 1] 标量:

Q = 0.40 · url_coverage + 0.40 · checklist + 0.20 · spec

权重选择反映了一种价值判断:前两项 url_coverage 和 checklist 才是 DR 智能体真正的产品价值,即「找对了页面」和「满足了任务要求」这两件事同样重要,因此各占 0.40,平起平坐;spec 是底线,只占 0.20,意思是「报告不能是空白,但只要不是空白就不再加分」。

注意一个关键事实:Q 里只用了 7 项 pillar 中的三项 (url_coverage、checklist、spec),其余四项 (url_reachability、quote_match、citation_alignment、analysis_depth、presentation,以及已弃的 claim_nli) 暂时不在 Q 里。这是有意的:头部 v2 公式之所以能让真实性门反转排名 (见 §11.1),恰恰是因为我们没有把 url_reachability 折进 Q,而是把它单独拎出来乘到 Q 外面,使它从「占一份权重」升格为「能把整个分数清零」。具体怎么乘,看下面三个公式。

三个公式

composite_v1 是最简单的形式,它直接令复合分等于质量项,不引入门:

composite_v1 = Q

这个公式刻意把 url_reachability 排除在外,作为「没有真实性门」的对照组。一个 URL 全是捏造的智能体在 v1 上仍能拿到 Q 那么高的分数,只要它的引文模式凑巧覆盖到一些黄金 URL、checklist 判官又被它的文笔骗过,v1 给它的数字会非常体面。这正是 v1 存在的意义:它让我们能定量看到「不带门的公式会奖励什么」。排行榜里 v1 数字不参与头部 Elo,但和 v2 并列展示,反转的现象 (§11.1) 就是直接对比 v1 与 v2 的名次得到的。

composite_v2_truthful 是头部排名实际使用的公式,它把 url_reachability 乘在 Q 外面:

composite_v2_truthful = url_reachability × Q

乘性的含义非常硬:如果一个智能体的可达性是 0,意味着它引的所有 URL 都打不开,那么不管 Q 多高,复合分一律乘 0 = 0。如果可达性是 0.5,意味着只有一半的引文落地,那么 Q 直接减半。如果可达性是 1.0,意味着所有引文都能打开,Q 才能不打折地传递到复合分。「捏造引文」在这个公式下从「扣分项」升格为「归零项」,这是 v2 在方法学上的核心选择。头部 Elo 表上每个智能体的 Elo 数字,背后都是用这一个公式对每个任务算出来的复合分进行两两对战聚合而成。

composite_v3 是一个补充实验:它扩展到 7 个 pillar (把 quote_match、citation_alignment、analysis_depth、presentation 也加进来),同时把硬门松弛为「有下界的门」,下界取 0.1:

composite_v3 = max(0.1, url_reachability) × ( 0.20 · url_coverage + 0.20 · quote_match + 0.20 · checklist + 0.10 · spec + 0.15 · citation_alignment + 0.10 · analysis_depth + 0.05 · presentation )

v3 括号里的七个权重相加正好是 1.00 (0.20 + 0.20 + 0.20 + 0.10 + 0.15 + 0.10 + 0.05),这一项可以理解为「7 维平均加权后的质量项 Q′」,然后外面再乘一个下限不会低于 0.1 的可达性。下界 0.1 让捏造智能体不再字面上拿 0 分,而是拿到 Q′ 的十分之一。v3 的价值不是替代 v2,而是提供一个更柔和的、纳入了文笔与对齐的口径,使审计页可以排序「在 v2 上被门归零、但报告非空」的智能体,让它们不至于在 Elo 上彼此完全平手。v3 不驱动 Elo,Elo 仍然只看 v2。

完整走一遍:一份具体报告

这一节用一份虚构但典型的报告把上面三套公式从 7 个 pillar 数走到三个最终复合分,完整看一遍流程。假设报告打完所有验证器后得到:

pillar分数这一项说了什么
url_coverage0.75引用范围覆盖到该引 URL 的 75%
url_reachability0.90引的 URL 里 90% 真能打开
quote_match0.6060% 的引文位置在页面文本里有词重叠
checklist0.71任务 21 项中通过 15 项
citation_alignment0.55判官认为 55% 的引文真支撑论断
analysis_depth0.50跨源综合中等
presentation0.80结构清晰
spec1.00字数 / 引文数 / 段落数规范全过

合成的第一步是算质量项 Q。带入头部权重 0.40 / 0.40 / 0.20:

Q = 0.40 × 0.75 + 0.40 × 0.71 + 0.20 × 1.00 = 0.300 + 0.284 + 0.200 = 0.784

这 Q = 0.784 本身就是 composite_v1。它代表「不考虑 URL 是不是真能打开」时,这份报告在「引文范围 + 任务完成度 + 规范底线」上拿到的综合质量。

接下来把可达性 0.90 乘到外面,得到 composite_v2_truthful:

composite_v2_truthful = 0.90 × 0.784 = 0.706

因为可达性 0.90 是接近上限的,Q 几乎不被门稀释,从 0.784 只降到 0.706。这是「绝大多数引文都真存在」时门的温和效果。

最后算 v3。先把括号里的 7 维加权和算出来:

0.20 × 0.75 = 0.150 (url_coverage) + 0.20 × 0.60 = 0.120 (quote_match) + 0.20 × 0.71 = 0.142 (checklist) + 0.10 × 1.00 = 0.100 (spec) + 0.15 × 0.55 = 0.0825 (citation_alignment) + 0.10 × 0.50 = 0.050 (analysis_depth) + 0.05 × 0.80 = 0.040 (presentation) ───────────────────────── = 0.6845 composite_v3 = max(0.1, 0.90) × 0.6845 = 0.90 × 0.6845 = 0.616

这份报告最终在 score JSON 里就会写下三个数字:composite_v1 = 0.784、composite_v2_truthful = 0.706、composite_v3 = 0.616。三个数都对外可见。Elo 对战只看 0.706 这一个数,其它两个数留作审计与对照。

同一份报告,把可达性归零会发生什么

为了直观看到三个公式的差别,把上面同一份报告改成「文笔结构都还在,但 URL 全是捏造的」,也就是 url_reachability = 0,同时 url_coverage 因为 URL 错位会从 0.75 跌到 0.30 (它仍能蒙到少量黄金 URL,因为大量捏造里总有几个凑巧命中)。其余 pillar 几乎不变。重新算一遍:

Q = 0.40 × 0.30 + 0.40 × 0.71 + 0.20 × 1.00 = 0.120 + 0.284 + 0.200 = 0.604 composite_v1 = Q = 0.604 仍然不低 composite_v2_truthful = 0.00 × Q = 0.000 硬零 composite_v3 = max(0.1, 0) × raw_v3 = 0.10 × 0.575 = 0.0575 几乎零但非零

这一组对比就是头号发现的数学起源

同一份「文笔好的捏造」报告,v1 给它 0.604,v2 给它 0.000,v3 给它 0.057。三个数字进 Elo 后会让这份报告在三个公式下排到三种完全不同的名次。真实性门反转排名 (§11.1) 那个头号发现,本质上就是这一个数值反差在 9 个智能体、56 个任务上的累积。v1 名次让 gpt-researcher 登顶,v2 名次让它跌到第 5,差别完全来自上面这个 0.604 vs 0.000 的乘性门。

关于文字质量本身的分数

这里值得专门说一下,因为是常被问到的一点。读到上面的公式可能会发现:头部 composite_v2_truthful 里没有任何一项是测「报告本身写得好不好」的,Q 里的三项都是关于引文与任务完成度的,没有一项是关于「文字流畅性、结构组织、综合分析」。是不是说我们根本没评测文字质量?

不是,有专门的两项,但它们没进头部 v2。测文字质量的是 presentation (§6.7) 与 analysis_depth (§6.7) 这两个 pillar。presentation 一共 12 项细则 (6 确定性 + 6 LLM 判官),测的是标题层级、节平衡、列表表格、可读性指数,以及由判官判断的逻辑流、过渡质量、是否有元叙述废话、格式是否一致、结论是否综合、语气是否专业;analysis_depth 一共 10 项 (4 确定性 + 6 LLM),测的是跨源段落是否存在、矛盾语言是否被使用、三角证据是否被组织,以及由判官判断的是否超越枚举、是否做了因果推理、是否做了信息源三角、是否有新见解、是否有反驳意识、是否给出可行动结论。这两项每次运行都会被独立打分并写进 score JSON,在审计页可以单独排序。

它们没进头部 v2 是有意的方法学选择。原因很直接:如果把「专业语气」「逻辑流」与「URL 可达性」放到同一根坐标轴上做加权和,然后再决定排名,真实性门反转的现象就看不到了。一个文笔好但 URL 全是捏造的智能体 (典型代表 gpt-researcher 在任务 0009 上的报告),在「文笔分」上会拿满分,在加性合成里这部分会把可达性归零的损失补回来不少,排名上仍然名列前茅。我们恰恰要让「捏造引文」从这种补偿里跳出来,变成一个无法被任何其他维度救回来的硬归零信号。因此头部 v2 主动把这两项 pillar 排除在 Q 之外,让 v2 排名严格反映「引文是否真实 + 引文范围是否对 + 任务要求是否完成 + 报告形态是否过基本规范」这四件事的乘积。文笔好不好这件事,我们认为它非常重要,但它不该和「URL 是不是编的」放在同一根加权坐标轴上换算。

composite_v3 是另一种态度。它把这两项文字质量分纳入进来:7 维加权和里 analysis_depth 占 0.10,presentation 占 0.05,合计 0.15,也就是说 v3 复合分里大约 15% 的权重直接来自「报告本身的文字质量」。再加上 citation_alignment 也带 0.15 权重 (它本质上也算「引文与文字的对应」),v3 里相当于 30% 是「文字侧」的评分。v3 的存在,正是为了给「想把文字质量也综合考虑」的读者一个可对照的口径,它每次也都被计算并对外可见,但它不驱动头部 Elo。

简单收一句:文字质量我们测了,而且测得很细 (12 项 + 10 项,共 22 个细则项),只是在头部排名上故意没让它和真实性门混在一起。审计页和 v3 排名才是看文字质量的入口。

整条链条的总结

把这一节从头串一遍:每份报告先在 7 个维度上打分,产出 7 个独立的 pillar 数;其中 3 个 (url_coverage、checklist、spec) 按 0.40 / 0.40 / 0.20 加权得到质量项 Q;Q 本身就是 composite_v1;把 url_reachability 直接乘到 Q 外面得到 composite_v2_truthful,也就是头部排名用的那一个数;把 7 个 pillar 都按一组新权重 (含 analysis_depth 与 presentation 这两项文字质量分) 相加再乘上 floor-0.1 的可达性,得到 composite_v3,提供更柔和、纳入文字质量的对照口径。每个任务上,任何两个智能体的 v2 谁高谁就赢这一场对战;所有任务上所有两两对战累积起来,送进 Bradley-Terry / Elo (下一节 §8 讲),就得到了头部排行榜上的那 9 个 Elo 数字。

8. Bradley-Terry 排名

8.1 模型

排名聚合自每任务的 227 个成对对战 (storm 仅在 17 任务上有有效打分,贡献 136 对战)。对战由 composite_v2 合成:对每任务、每对都产出非退化打分的智能体,复合分高者赢、低者输,|Δ| < 0.005 记为平 (build_deep_leaderboard.py:241-244)。

tie-epsilon 取得很紧的原因:在 v2 下许多智能体塌缩到 composite=0 (可达门把它们杀了),loose epsilon 会让这些智能体彼此平,而非按处理顺序决出胜负。

Elo 更新

E_a = 1 / (1 + 10^((R_b - R_a) / 400))
R_a' = R_a + K · (s_a - E_a)

K = 32,起始 1000,跑 20 轮,每轮 shuffle 对战次序,最后取 20 轮均值 (src/scoring/arena.py:84-120)。多轮平均是为了消除单轮 Elo 在小数据上对对战顺序的敏感性。

Bradley-Terry 似然

L(π) = ∏ (π_i / (π_i + π_j))^w_ij , R_i = 400 · log₁₀ π_i + C

其中 w_ij 是 i 击败 j 的次数。Elo 多轮平均可视作上述似然 MLE 的迭代近似。

8.2 Bootstrap 置信区间

非参数 bootstrap:对全部对战做 B = 1000 次有放回重采样,每次重新拟合 Elo,取 2.5% / 97.5% 分位数。头部排行榜 CI 半宽介于 ±43 与 ±96 之间;窄者 (例如 storm 的 ±43) 是参与对战数较少的智能体。

8.3 置换 p 值显著性检验

成对置换 (N = 1000):在相邻名次对内 shuffle 对战结果,问「两个智能体等强」的零假设下观察到至少这么大 Elo 差距的概率是多少。报告 §10 给出每对相邻名次的 p。

9. 退化样本怎么过滤

把退化样本纳入 Bradley-Terry 会扭曲评分。Runner-error placeholder (例如 (qx-agents error: ValidationError ...)) 是一个字符串,打分器愉快地抽不出 URL,LLM 判官也愉快地在近乎空答案上输出 PASS 或 FAIL。最终复合分 (~0.02) 既不算赢任何人,也不算输给任何人。过滤器在 Bradley-Terry 之前剔除它们 (build_deep_leaderboard.py:82-131)。四种模式:

#模式条件
1短答案 + 无信号answer_chars < 600 且 url_reachability.score = 0 且 checklist.pass_rate = 0
2沙盒基础设施失败url_reachability.details.infra_failure == True。每次探针都是 5xx 或 net-fail,意味着沙盒打分期间挂了。
3判官全失败checklist.judge_error 且 analysis_depth.judge_error 且 presentation.judge_error。三个 LLM 判官全错 (通常是 ds_proxy 超时),没有判官信号。
4Runner failure / exception 占位答案 Markdown 以 (<Agent> produced no report after Ns, exit=N) 或 (<Agent> error: ...) 开头

排行榜审计行专门保留了被过滤智能体的元信息,以防它们「未被基准测试过」式地静默消失。

具体效果:354 个打分文件中 326 个被保留进 Elo,28 个被过滤。其中 qx-agents 30/30 全被过滤 (SDK 框架不兼容),storm 13/30 被过滤,flowsearcher-ds 18/48 被过滤。

支持「带门」公式的最强论点

一个读起来很美、引文却是捏造的 DR 智能体,比一个文笔笨拙但引用真实页面的智能体更糟糕,因为流畅捏造在下游被相信的概率更高。真实性门让这一点直接可见。

11.2 多智能体协作架构在本沙盒上占优

头部三框架中,两个是多智能体协作 (camel-ai 的 ChatAgent + SearchToolkit、flowsearcher-ds 的记忆引导规划器 + 执行器),一个是受约束的工具调用循环 (smolagents 的 ToolCallingAgent + max_steps=60)。底部六个中,有两个是不分解的单循环或顺序流水线 (ii-researcher 的 ReAct 在 3-5 轮后终止;gpt-researcher 的 writer 是单次)。

趋势与下面这条假设一致:DeepSeek V4-flash 作为非推理骨干,得益于外部分解。当框架把任务分解为子查询并编排它们时,模型能把每个子槽填好;当框架要求模型在一个提示里隐式分解时,输出要么截断要么漂移。

这个发现以沙盒为界:120 引文下限正是强制分解的来源。DeepSeek V4-flash 的一轮前向几乎不可能在单次搜索之后产生 60 条跨源引文,因此任何不在多轮搜索中重试的框架,结构上就被卡在引文下限以下。换用推理型骨干 (DeepSeek V4 thinking on 或 GPT-5 reasoning) 时,多智能体与单循环之间的差距可能收窄。

11.3 社区热度 vs 实测排名分歧

gpt-researcher 在本基准上排名第 5,但其 GitHub 热度 25.7k stars,是开源世界中「deep research」一词的事实参考实现。它的排名落后于一个大多数读者没听说过的框架 (flowsearcher-ds,自研) 与一个并非专为 DR 而设的通用智能体循环 (smolagents)。这一分歧有三层成因:

① 热度选择「产品级实现」

gpt-researcher 是面向开放网的优秀工程:提示词针对真 Google / Tavily 调过,检索流水线假设真 CDN 提供 HTML,报告器期待开放网 URL 模式。把它放到一个 Magento 沙盒上 (URL slug 含数字后缀,模型无法模板化),它的工程优势不可迁移。

② 热度选择「文笔好」

大多数使用者读完报告就完事,不会去点 93 个 URL 看有几个 200。gpt-researcher 文笔好,以文笔评价 DR 的社区做的事情,正是 composite_v1 做的事。composite_v2 就是把差异显化的工具。

③ 热度选择「容易上手」

gpt-researcher 的一行启动是真本事;smolagents 是没有任何 DR 营销光环的通用工具调用循环。易上手与「实测质量」相关性弱。

启示不是 gpt-researcher 不好,而是「社区最爱」与「在密封沙盒上的实测质量」是两根不同的坐标轴,而本基准的存在就是为了刻画第二根。

12. 局限性与公平性审计

12.1 五个已知局限

三个语料,而非整个网络

Magento + Postmill + Kiwix 覆盖了电商、社区、百科三种垂直模式,但没有新闻、学术文献、原始文档,以及登录墙背后的一切。在「金融报告合成」上优秀的智能体不会被本基准测出。代价是有意的:能覆盖整个开放网的沙盒就是开放网。

头部数字使用单一骨干

头部排名所有智能体都跑在 deepseek-v4-flash 非推理上。风险:用 DeepSeek 排出来的名次告诉我们「哪个智能体架构对 DeepSeek 最合适」,而非「哪个智能体架构最通用」。/compare 页与 ds_proxy 可切后端正是为此而设。

LLM-as-judge 偏差

7 个维度中有 3 个 (checklist / citation_alignment / presentation+depth 的 LLM 部分) 都调用同一份 DeepSeek V4-flash 判官。确定性守卫 (三种) 捕获了最坏情况,但残余偏差非零。未来迭代应换用不同家族的判官 (GPT-5、Claude) 测量一致性。

部分任务覆盖

100 个设计任务中只有 57 个完成端到端打分。若全部 100 个都跑,排行榜 Elo CI 半宽会从 ±43~±96 收紧到约 ±30~±60。

进程内 runner 虚拟环境冲突

三个进程内 runner (camel-ai、smolagents、flowsearcher-ds、deepagents 部分) 共用 .venv-camel。任一框架的依赖升级可能掩盖另一框架的依赖冲突。子进程 runner 每个独占一份虚拟环境,避开了该问题。未来清理是把每个框架都收进自己的虚拟环境并用 clean runner 启动。

第六个 (较小的) 局限:头部 composite_v2 自 2026-04-27 起弃用 claim_nli (乘数 1.0),原因是 NLI 判官在视觉噪声大的商品页上不可靠。原则上的修法是换非 DeepSeek 的 NLI 判官或手工校准。

12.2 公平性审计

审计结论 (data/results/audit/HUMAN_URL_AUDIT.md 与 final_audit.json):14 个被接入的框架中,审计员人工抽检每个框架的代表性运行,核对「智能体被打分文件里的 URL 是否真的出现在该智能体的报告 markdown 中」与「报告 markdown 是否就是该智能体真正的产物 (而非中间体或被截断)」。14/14 框架 0 个存在我们这一侧的失误 (Our-Fault)。 所有归零的运行都是智能体侧失败 (要么捏造 URL、要么超时、要么框架结构性 bug),不是我们打分链路的失误。

14.1 方法学相关

为什么不在 GAIA / BrowseComp / AssistantBench 上跑就行?
三个理由。① 它们是开放网,页面会漂、404、被重排,实验不可复现。② 真实性几乎不可探测:同一 URL 从不同主机看,返回码可能不同。③ 它们把「产品评测」和「能力测量」混在一起,而我们要测的只是后者。我们沿用了 ALCE / WebArena 的「先冻结基底再打分」思路,把它推广到多语料的交互式沙盒。
为什么真实性门要乘性而不是加性?
加性会把「文笔好的捏造」奖励上去。一个 reach=0 的智能体在事实上没有产出可用证据,加性公式下它仍然能靠 checklist 与 url_coverage 拿到大半分。乘性硬零是「捏造不可妥协」最简单的表达;v3 的 max(0.1, reach) 是为了让审计页可看到 v2 上归零但报告非空的智能体,但头部排名仍用 v2。
url_coverage 和 url_reachability 信号重复吗?
不重复,两者刻画不同失败模式。coverage 问「你引用的 URL 是不是该引的那些」,reachability 问「你引用的 URL 是不是真的存在」。gpt-researcher 是个反例:它的 coverage Elo 全场最高 (1305) 但 reach Elo 倒数第三 (883),意思是它输出的字符串很多看起来像该引的 URL,凑巧也匹配了一些黄金条目,但真去 HTTP 拉就拉不到。两个 pillar 互不可替代。
LLM 作判官有自我偏好/松懈/假阳性怎么办?
三层守卫。① 退化答案守卫:50 词以内 + 无引文直接全 FAIL,不调 LLM。② 全 PASS 短答案降级:全 PASS 但 <500 词全 FAIL,捕获松懈模式。③ 锚定关键词交叉检查:reach <0.30 时含锚定关键词的项降级 FAIL。这三层让 gpt-researcher 在引 93 个伪 URL 的报告上拿不到 21/21。残余偏差靠未来的多判官族 (GPT-5、Claude) 一致性比较来量化。
为什么 claim_nli 从 v2 弃了还要继续算?
为了可追溯。DeepSeek V4-flash 在视觉噪声大的商品页上假阴性多,我们 2026-04-27 把它在 v2 中乘数置 1.0。但它仍每次运行被计算并持久化在 score JSON 里,以便未来换更强 NLI 判官时,可以重跑头部数字而不重跑智能体。原则上的修法是换非 DeepSeek 的判官或手工校准。
头部 v2 公式里没有任何「文字质量分」,是不是没评测报告写得好不好?
评测了,而且测得很细,只是没让它进 v2。文字质量评分集中在两个 pillar:presentation (12 项:标题层级、节平衡、列表表格、可读性 + 6 项 LLM 判官 ── 逻辑流、过渡质量、无元叙述废话、格式一致、结论综合、专业语气) 和 analysis_depth (10 项:跨源段落、矛盾语言、三角证据 + 6 项 LLM ── 超越枚举、因果推理、信息源三角、新见解、反驳意识、可行动结论)。两项共 22 个细则项,每次运行都被独立打分并写进 score JSON。它们在头部 v2 中被主动排除,因为把「专业语气」与「URL 可达性」放在同一根加权坐标轴上做加法,会让捏造但文笔好的报告靠文字分把可达性损失补回来,真实性门反转就看不到了。composite_v3 是另一种态度,把这两项纳入 (depth 0.10 + presentation 0.05,合 0.15 权重),提供「想综合考虑文字质量」时的对照口径,但仍不驱动头部 Elo。简单一句:文字质量在审计页和 v3 排名看,头部 v2 故意不让它和真实性混在一根轴上。

14.2 实验/结果相关

9 个框架排名,统计上真的有区分度吗?
部分有,部分没有。头部三名 (camel-ai / flowsearcher-ds / smolagents) 是一簇,后六名是一簇,中间隔一条 362 Elo 的鸿沟 (p < 0.001)。其余相邻名次的置换 p 值都 > 0.30。这表明正确读法是「两个簇,一条鸿沟」,而不是「9 个有序档次」。
flowsearcher-ds 是自研的,会不会有偏?
三件事。① 它跑在和其他 12 个框架完全相同的 shim + ds_proxy 上,没有特殊权限。② 它的代码与其他 runner 一样在 scripts/run_flowsearcher.py 透明可读。③ 它没有访问黄金池 (只有打分器看得到)。它的优势来自记忆引导规划 (L1/L2/L3),具体表现是尾部更窄 (近 0 的运行更少,均值 v2 = 0.296,低于 camel-ai 的 0.410,但成对 Bradley-Terry 奖励的是「赢得多」而非「均值高」)。审计页对它和其他框架用同一份判官,我们没有为它写特殊守卫。
为什么 qx-agents 被排除而不是低分?
因为它的失败是结构性的。在 DeepSeek V4-flash 下,openai-agents SDK 抛 pydantic.ValidationError: 2 validation errors for KnowledgeGapOutput,30 个任务全是同一个 stack trace,placeholder Markdown 形如 (qx-agents error: ...)。把它当作 0 分进 Elo 会扭曲 BT 评分:它「输」的方式不是智能体表现差,而是 SDK 根本没跑。退化过滤器 (#4 模式) 把这种 placeholder 字符串前缀剔除,但审计行公开记录了它的存在,所以它不是「未被基准测试过」式地消失。
storm 修一个 bug 就涨这么多,是不是修过头了?
不是。修复内容是 storm_runner.py:299-313 在 Markdown 尾部追加一段从 url_to_info.json 重建的 References 节。这一段不是新生成的内容,它是 STORM 自己已经计算好的 URL 表,我们之前打分时没合并回来,等于打分了一份被剥引文的正文。修完后 composite_v2 从 0.000 涨到约 0.067,反映的是「之前在测错的产物」。STORM 仍然第 9 名,只是从「全场归零」变成「能参加比赛」。
为什么 gpt-researcher 不直接 reach 修一下就行?
它的问题不是配置,是架构。它通过把商品名 kebab-case 化为 /products/<slug> 来构造 Magento URL,但 Magento 真实 slug 含智能体看不到的数字后缀 (例如 /sennheiser-hd650-headphones-12345.html)。模型没法猜数字后缀,所以构造的 URL 几乎全部 404。修法只能是改它的 retriever,让它从 Tavily 搜索结果里 copy 真 URL,而不是构造,这是一个上游框架级改动。本基准的处置是不修上游,但忠实地把这个失败显示给读者:per-pillar 表里 gpt-researcher 的 url_coverage 1305 + reach 883 就是这个模式的指纹。

14.3 工程/复现相关

这套基准能跑在多大硬件上?
最低配置:64 GB RAM + 200 GB 磁盘 (Magento 镜像 + Postmill + ZIM 文件)。GPU 不是必需的 (所有 LLM 都走 DeepSeek API),但有 GPU 可以本地跑判官省钱。评测主机是 Windows + WSL Ubuntu 一台,Magento + Postmill + Kiwix 都是 docker container,shim + ds_proxy 是常驻 Windows 计划任务。复现一份 30 任务的小基准约 6-8 小时墙钟,$15-30 的 DeepSeek 调用。
为什么用 13 个虚拟环境?
框架依赖冲突。gpt-researcher 依赖 pre-1.0 LangChain 子模块 (langchain.docstore、langchain.vectorstores),而 langchain-odr 用的是 1.x。ldr 用 Python 3.12 程序化 API。tongyi 用 qwen_agent。把它们装到一个 venv 里要么装不上,要么互相覆盖。子进程范式 + 每框架独占 venv 是已知的代价,但回报是接入第 14 个框架不会炸前 13 个。
为什么搜索 shim 要 Tavily 兼容?
因为每一个被接入的框架都已经支持 Tavily。Tavily 是 DR 框架社区事实标准的搜索 API,有清晰的 request/response schema。我们写一个 Tavily 兼容的 HTTP 服务,每个框架的接入就只是改一行 base_url 配置,无需自定义工具或调提示词。这是降低接入成本最关键的设计决策。换 Brave / Serper / 自定义 schema 都意味着改 13 个框架。
为什么 ds_proxy 也要 OpenAI 兼容?
同样的理由。每个 DR 框架都用了 openai Python SDK 或类似客户端。OpenAI 兼容代理意味着改一行 base_url。代理在请求体里注入 thinking: disabled,确保所有框架都跑在同一非推理 checkpoint 上,这是「测量智能体架构而非骨干强度」的前提。
Web 应用为什么没用数据库?
够用就行。排行榜 JSON 大约 100 KB,任务级 score JSON 总共几 MB。FastAPI 在每次请求时读盘,无数据库、无 Redis、无后台刷新。bulk runner 写下新 score JSON,等下一次 build_deep_leaderboard.py 跑完,/api/leaderboard.json 自动看到新结果。这套「实时读盘」对一个分钟级刷新的小基准就够了。复杂度收益不正。

14.4 哲学/边界相关

真实性门反转,会不会只是「门过严」的人为产物?
不是。两个证据。① per-pillar Elo (§10.3) 表明 gpt-researcher 在所有可达性以外的维度都表现良好 (checklist 1152、url_cov 1305、spec 1238),它输的是单一维度 reach 883。② v3 的松弛门 (floor 0.1) 给捏造智能体 10% 权重,结果仍然让 camel-ai 在头部、gpt-researcher 在尾部,反转模式持续。说明这不是「v2 太严」的产物,而是「真实性是一根独立坐标轴」的产物。
沙盒只有 2000 个商品页,智能体会不会「背下来」?
理论上可以,实际上没有。① 智能体在评测时是新开的 Python 进程,没有跨运行状态。② Tavily shim 的响应是 BM25 排序的标题片段,智能体看不到完整页面字典。③ 黄金池中 must_cite_urls 120+,不可能在上下文里塞完。④ 即使能背,reachability 探针仍然检验 URL 是否真实存在,不存在的 URL 哪怕模型再确信也不算分。背诵失败的可能性在沙盒里反而比开放网低,因为我们能精确检验。
结论能外推到开放网的 DR 场景吗?
部分可以,部分不能。可以的:① 「带真实性门 vs 加性」结论是公式选择,与基底无关。② 「多智能体协作在非推理骨干上占优」与骨干 / 任务结构有关,与基底关系小。不能的:① gpt-researcher 在开放网上的真实排名我们不知道,只知道它针对开放网工程化得很好,搬到 Magento 会失去工程优势。② 沙盒里 reach 0.30 阈值在开放网上 (考虑 CDN 抖动 / 限流) 可能要松一些。我们没声称要外推,只声称要刻画「在固定基底上的智能体逻辑」。
这个工作的真正贡献是什么?
不是 Elo 数字本身。真正的贡献是:用一个可被任何人重建的沙盒,证明了「以加性方式打分会让 DR 评测奖励错误的行为」。这是一个方法学命题,在开放网基准上无法证实 (因为真实性不可探测),也很难证伪。把它证实是这个工作的核心。其余 (13 框架适配、7 维评分体系、Bradley-Terry + Bootstrap + 置换、Web 应用) 都是支撑这个命题所必需的工程,但不是命题本身。

15. 评分体系的局限与扩展设计

这一节回答一个尖锐但重要的质疑:我们的 7 个 pillar 是不是太集中在「URL 引用」这一件事上了?报告的其它属性,比如说具体数值正不正确、报告内部前后说法是否一致、是不是只展示了一面之词,这些维度在当前体系里几乎没有覆盖。这一节先把别人怎么打分调查清楚,再把我们的盲区诚实地标出来,最后给出一个具体的扩展设计。

15.1 这个质疑值得严肃对待

批评本身是成立的。把 7 个 pillar 按主题归一下:url_coverage / url_reachability / quote_match / citation_alignment / claim_nli 这五项都围着引文转,checklist 是任务专属完成度 (与引文模式高度相关),analysis_depth 与 presentation 才真正与「报告本身写得怎么样」有关。换句话说,我们的 7 项有 5 项在测同一件事(引文是否真实、是否对应),2 项在测文笔与综合性,然后就完了。没有一项专门测「报告里说的具体数字是不是对的」、「报告自己内部有没有自相矛盾」、「报告有没有给出多个视角」。这三个轴在文献里其实是有独立指标的,我们没用上。

这一节不是给体系打补丁式辩护,而是承认 7 pillar 没覆盖完然后提出怎么补。这是答辩时如果被问到「你们怎么只测 URL?」最干净的回答:我们测了 URL 之外的事,但承认还有三个轴没测,这是下一版要做的。

原子事实正确性❌不验证「价格 $349」「评论数 1234」「电池 30 小时」这类数值是否与源页一致
内部一致性❌「第 3 节说 Sony 降噪最好,第 7 节说 Bose 降噪最好」我们抓不到
视角/客观性❌对一个一边倒夸 Sony 不提缺点的报告,我们没有惩罚机制
推理验证 (硬)⚠ 部分有「analysis_depth.causal_reasoning」LLM 项,但不提取论证链、不单独评分
实体去重/停止判断❌智能体「什么时候算搜够了」我们没测

简单收一下:引文层我们做得最好,文字层也覆盖了,但「数值对不对」「报告内部说话一致不一致」「视角是否多元」 这三件事完全没测。这正是用户的直觉。下面给出补救方案。

15.4 扩展提案:四个新 pillar

不打算一次加 5+ 项,因为 LLM 判官每多一项就多一份脆弱性。优先补四个性价比最高、文献最成熟、能在沙盒里做确定性实施的轴。其中一项 (source_diversity) 可以做到完全零 LLM 调用,这是文献调研后认为最值得先做的项,因为它没有判官脆弱性。

factual_exactness — 原子事实正确性

测什么。报告里所有可结构化提取的事实是否与被引源页一致。重点抓数值 (价格、星级、评论数、电池寿命、传输速率、年份、版本号)、命名实体 (品牌、产品名、人名、地名)、关系 (「Sony 收购了 X」、「Y 由 Z 在 2019 年发布」)。这是 FactScore / SAFE / VeriScore 在做的事,我们把它收缩到沙盒可结构化的范围内。

怎么做 (沙盒确定性优先)。分两步:

(1) 原子事实提取。对报告每段调一次 LLM (零温度) 输出 JSON 列表,每项含 {subject, predicate, object, source_url, span}。VeriFastScore 给我们一条优化路径:用一个小模型 (Llama 3.1 8B fine-tune) 同时做提取 + 验证,6.6× 速度。但第一版我们用 DeepSeek。

(2) 逐条验证。对每条原子事实:抓取 source_url 页面 (复用 reachability 的 fetch 缓存),先用正则确定性地查找声明的数值/实体 (例如:商品页价格用 $\d+(\.\d+)? 抽,直接 string contains;星级用 schema.org 微数据抽);确定性匹配不到的退化到 NLI 判官二次校验。

得分。verified_facts / total_extracted_facts,[0, 1]。

为什么沙盒里能做得比开放网好。Magento 商品页有结构化 schema.org 微数据,prices/stars/ratings 都在固定 XPath 上;Kiwix 维基有 infobox 抽取得到的固定字段。这两点让我们能把至少 60% 的原子事实用 regex 校验,只剩 40% 走 LLM 判官。开放网上做不到。

文献依据:FactScore (Min EMNLP 2023);SAFE (Wei 2024);VeriScore (Findings of EMNLP 2024);VeriFastScore (arXiv 2505.16973);FaStFact (arXiv 2510.12839)。

internal_consistency — 报告内部前后是否自相矛盾

测什么。同一份报告里,前后两处对同一实体的论断,是否相互矛盾 (而非互补)。例:第 3 节说「Sony WH-1000XM5 在 ANC 上击败 Bose QC45」,第 7 节说「Bose QC45 的 ANC 是同价位最强」,这是矛盾;但「Sony 续航更长 + Bose 重量更轻」不矛盾。

怎么做 (SummaC + ContraGen 路径)。三步:

(1) 句子切分 + 实体聚类。把报告切成句子,对每句抽出主体实体 (NER 或 LLM)。把含同一实体的句子聚成簇。

(2) 簇内成对 NLI。对每个簇内任意两句 (s_i, s_j),跑 NLI 判官二选一:「entail / contradict / neutral」。文献教训:NLI 单独做跨句矛盾会塌到 41% (近似随机) ── 必须 hybrid。我们用 SummaCConv 的句对聚合 + LLM 判官二次复核 contradict 标的对的方案。

(3) 得分。1 - (contradicting_pairs / total_same_entity_pairs)。

关键优化。不对全文 N² 对做 NLI (代价爆炸),只对同簇内对做。典型报告 200 句、20 实体簇、每簇 10 句 → 20 × C(10,2) = 900 对,可控。

这一项捕获什么真实失败。合成长报告的 DR 智能体在 6000 词以上经常前后矛盾,因为不同 paragraph 是不同 search 子线程的产物。当前我们的 7 pillar 完全测不出这种失败。

文献依据:SummaC / SummaCConv (Laban TACL 2022);ContraGen (arXiv 2510.03418v1, 2025-10);Contradiction Detection in RAG (arXiv 2504.00180, 2025-04);A Straightforward Pipeline for Targeted Entailment (arXiv 2508.17127, 2025-08)。

perspective_balance — 是否给出多视角

测什么。对于评价类、推荐类、比较类任务,报告是否同时呈现正面与负面视角?是否对每个被评价实体给出 pros 与 cons?是否在中立任务上保持中立、不偷塞主观推断?DRACO 把这一轴塞在 presentation 维度下的 objectivity 子项。

怎么做。分两层:

(1) Tier A 确定性 (情感词典)。用一份 ~500 词的双向情感词典 (positive/negative 各 250 词,英文,VADER + 手工 DR 域增补),对每个被评价实体,在其周围 100 字符窗口内计正负情感词频。要求每个被推荐实体至少有 ≥ 1 negative 词;过度偏一方 (比例 > 5:1) 计负。

(2) Tier B LLM 判官。对每个被推荐实体,让判官回答「这份报告是否对该实体给出至少一条具体且非琐碎的局限/缺点?」 二值。

得分。0.4 · tier_a + 0.6 · tier_b,[0, 1]。

沙盒友好性。Postmill 论坛里我们已经预先标注了 5 类情绪 (praise / complaint / technical_question / comparison / purchase_advice),报告若不引用 complaint 类帖子,情感平衡会自然失分。

文献依据:DRACO 4-dimension rubric (Perplexity 2025);WikiNeutral NPOV evaluation;DREAM 的 writing quality 子维度含 objectivity。

source_diversity — 信源与视角的结构性多样性 完全确定性,无 LLM

测什么。报告引用的 URL 在信源层面是否多元,而非只挂在一个域、一种来源类型上。这一项与 perspective_balance 互补:perspective_balance 测「对每个实体是否给出正反两面观点」(语义层),source_diversity 测「报告整体的信源分布是否健康」(结构层)。

怎么做 (零 LLM)。四个确定性子指标:

  • 域熵 (domain entropy):H = -Σ p_i log p_i,其中 p_i 是引文落在该 host:port 的比例。三个沙盒域均匀分布则 H = log 3 ≈ 1.10;一边倒到一个域 H → 0。
  • Top-1 share:最大单一域的引文占比 ≤ 阈值 (典型 0.45)。防止「报告 95% 是 shopping URL」。
  • Source type 分类:用静态规则把每个 URL 归入 product_page / forum_post / wiki_article 三类 (沙盒里这三类对应三个 host),要求每类至少有最低占比 (典型 0.10)。
  • 论坛子版多样性 (沙盒友好)。Postmill URL 含 /f/<forum>/ 子版标识,要求引用至少 ≥3 个不同子版。

得分。0.35·H_norm + 0.25·(1{top1 ≤ θ}) + 0.25·type_balance + 0.15·subforum_diversity,[0, 1] 之间。整条公式纯 Python 计算,不打任何 LLM。

为什么值得做。(1) 实施成本几乎为零 (一个 verifier 文件 ~150 行);(2) 完全确定性,跑两次结果一样,无判官脆弱性;(3) 文献验证:LiveDRBench (Microsoft 2025) 整个基准都走「全确定性 precision/recall/F1」路线,DRACO 的 citation quality 维度子分含「来源多样性」。(4) 直接回击「你们太集中在 URL 引用上」的批评,因为它本身就是 URL 之外的结构性指标。

文献依据:LiveDRBench (Microsoft 2025);Vendi Score (Friedman & Dieng 2023);DRACO citation quality 维度 (Perplexity 2025);political communication 的 actor / viewpoint diversity 度量。

关于 reasoning_validity 我们暂不加的原因

论证链验证 (reasoning_validity) 是 DREAM 评测的 Reasoning Quality 轴,但 DREAM 本身没公开它的具体打分细则,我们目前能想到的方案都过度依赖 LLM 判官 (一条「A because B」论断要让判官判断 B 是否真支撑 A,这本身就是更难的判官任务)。一条更鲁棒的路径是 Toulmin / Freeman 论证模型:用正则识别因果连接词 (because / since / therefore / hence / due to / 因此 / 由此) 切分出 (premise, conclusion) 对,先用确定性守卫检查 premise 句是否有引文且 quote_match 通过,只对通过守卫的样本走 LLM 判官评估「premise → conclusion 是否逻辑有效」。这条路径成熟后再加。另一个不加的轴是「实体去重 / 停止判断」 (DeepSearchQA 的轴),它测的是「智能体应该什么时候停」,与「报告本身的质量」是两个不同问题,更适合放在智能体行为而非报告质量评测里。

15.5 重新设计的合成公式 composite_v4

把 4 个新 pillar 加入,重新分配权重,要求所有权重和恰为 1.00。头部仍然是「真实性门 × 质量项」结构,只是质量项的内部组成变了:

composite_v4 = url_reachability × ( // 引文层 (传承 v2/v3) 0.10 · url_coverage + 0.05 · spec + 0.10 · checklist + 0.10 · citation_alignment + 0.05 · quote_match // 信息正确性层 (新) + 0.13 · factual_exactness ← NEW // 内部一致性层 (新) + 0.13 · internal_consistency ← NEW // 视角层 (新) + 0.08 · perspective_balance ← NEW // 来源多样性层 (新,纯确定性) + 0.06 · source_diversity ← NEW // 文字层 (传承) + 0.10 · analysis_depth + 0.10 · presentation ) // 权重核对:0.10+0.05+0.10+0.10+0.05+0.13+0.13+0.08+0.06+0.10+0.10 = 1.00 ✓

四处权重再分配的逻辑:

  • url_coverage 与 checklist 从 v2 的各 0.40 降到各 0.10,因为它们在 v2 里占太重 (引文一项就占 50% 总分),稀释给信息层与一致性层。
  • 新加的 factual_exactness 与 internal_consistency 各拿 0.13,因为它们是真正未被覆盖的盲区,权重应该接近已有的引文 pillar。
  • perspective_balance 取 0.08,中等略低,因为它对非评价类任务不强相关 (枚举型、辟谣型任务不需要多视角)。
  • source_diversity 取 0.06。权重不是很大,但它是整个加权和里唯一一项零 LLM 判官、完全确定性的,可以视为「真实性门之外」的第二条不可作弊的硬轴。
  • 头部 url_reachability 仍作为乘性门,不变。这是真实性门反转那个核心发现的源头公式,v4 保留。

v4 下「文笔好的捏造」会变成什么

把 §7.5 那个 reach=0 的反例放到 v4 公式下走一遍:它的 reach 仍是 0 → 整个 v4 仍归零,头部判决不变。但同时,internal_consistency 与 perspective_balance 会让另一类失败浮现出来:「URL 是真的、但报告自相矛盾」或「URL 是真的、但只夸不批」。这两类失败在 v2 下可能拿很高分,但在 v4 下会被两项新 pillar 拉低。v4 不是替换 v2,它是把 v2 不能区分的两类「URL 真但报告还是有问题」分离出来。

15.6 路线图与代价

四项扩展不是同等优先级。按可落地性排序 (而非按权重),零 LLM 的项放最前面,LLM 重度依赖的项放后面:

顺序新 pillar实施成本风险预期收益
第一步 source_diversity 极低。~150 行 Python,完全 offline,零 LLM 调用。每报告 <1 秒。 无。确定性指标,跑两次结果一样。 直接拿到一根「URL 之外」的硬轴;一晚能上线;最干净地回击「我们只测引用」的批评。
第二步 factual_exactness 中等。需要原子事实提取 (~1 LLM 调用/段) + 60% 用正则确定性校验 (商品页 schema.org 微数据、维基 infobox)。每报告 ~20 LLM 调用,2-5 分钟。 低。FactScore 系列方法成熟,沙盒结构化数据帮我们把 LLM 比例压到 40%。 能直接抓到「价格抄错」「电池寿命瞎编」类失败,这类失败当前完全测不出。
第三步 perspective_balance 低。Tier A 是情感词典确定性,Tier B 是每实体一次 LLM 调用。每报告 ~10 LLM 调用,1 分钟。 低。情感词典是确定性的,Tier B 只问一个二值问题。 能抓到「一边倒推荐」类报告,对评价类任务相关性高,对其它任务有限。
第四步 internal_consistency 高。同实体聚类 + 簇内 NLI + LLM 复核。每报告 ~900 个 NLI 调用 (本地 DeBERTa-v3-MNLI 即可,无需 LLM) + ~20 个 LLM 复核调用。 中。NLI 跨句矛盾文献里报告精度低 (~41%),需要严格走 hybrid 路径,且本地 NLI 模型与 LLM 判官的边界需要校准。 能抓到 6000+ 词长报告的前后矛盾,这是头部三框架以外几乎所有 DR 智能体的常见失败模式。最值得做、也最难做对的一项。

成本估算

当前一份报告打分总 LLM 调用约 30 次 (checklist 1 + citation_alignment 50 + claim_nli 30 + analysis_depth 6 + presentation 6,部分能并行)。四项新 pillar 加起来约 +50 LLM 调用 / 报告 (source_diversity 0 + factual_exactness 20 + perspective_balance 10 + internal_consistency 20)。本地 NLI (DeBERTa-v3-MNLI) 不计成本但需要 GPU,RTX 5090 32GB 一秒 ~100 对足够用。LLM 用 DeepSeek V4-flash,每调用 ~$0.002,新增 ~$0.10 / 报告。56 任务 × 9 头部框架 = 504 报告 → 新增 ~$50 一次全量 rescore。可接受。

15.7 方法学上的额外改进 (不只是加 pillar)

把新 pillar 一项项往上加,只是横向扩展评分维度。文献里还有一组纵向改进,与维度无关,但能让现有体系的鲁棒性大幅提升。这些改进不要求改 pillar 的数量与权重,只要求改打分流程,在 v4 落地的同时一并做最合算。

判官族多样化 (judge ensemble) 最优先

我们目前 7-10 个 LLM 判官调用全部打到同一份 DeepSeek V4-flash。文献里反复出现的「判官与被评测模型同源会引入自我偏好」我们用三层确定性守卫缓解,但这不是根除。彻底修法是引入不同家族的二号判官 (GPT-5、Claude Opus 4.7、Qwen3.5 推理变体之一),对 LLM 重度维度 (citation_alignment、checklist、internal_consistency Tier B、perspective_balance Tier B) 做多判官投票。预算 +2× LLM 成本 (≈ +$100 / 全量 rescore),回报是给 LLM 判官加上可量化的误差棒。LiveResearchBench / DeepEval 用 Gemini 2.5 Pro + GPT-5 双判官取平均 + pairwise 时做 position swap 平均,这是 SOTA 做法。

参考报告锚定 (reference-based, RACE-style)

DeepResearch Bench 的 RACE 关键创新是 专家先写一份参考报告,judge 把它作为「该题该谈到什么」的锚点,然后用「自适应权重」决定四个高层轴的相对重要性。比纯 task-specific checklist 鲁棒,因为参考报告是具体内容,checklist 是抽象要求,具体内容能给判官提供更直接的比对锚。在我们 100 任务里挑 5-10 题难任务 (例如三个推荐锚点 0001/0002/0005) 由人工写一份示范报告,做 reference-based 加权打分,就能引入这层校准而不需要全 100 题都做。

失败模式分类法 (DEFT-style)

OPPO 的 FINDER + DEFT (2025-12, arXiv 2512.01948) 首次给出 DR 智能体的 14 类失败模式,横跨 reasoning / retrieval / generation 三大族。每份报告除了打数字分,还给一个失败类别向量 (例如:[hallucination=1, premature_stop=0, citation_drift=1, ...]),用于诊断而非排名。我们当前的退化样本过滤器 (4 模式) 是这个的简化版;扩展到 14 类后,审计页可以按失败类别筛报告,直接定位「哪个智能体最容易在哪种模式上出错」。这层信息对生态有用 (开源框架作者可以直接看到自己的失败画像),对 Elo 排名没有直接影响。

判官校准元评测 (RubricEval / JudgeBench)

RubricEval (arXiv 2603.25133)、JudgeBench、Prometheus 2 (2024) 给出「判官的判官」范式:跑一个小规模 meta-eval 集 (30-50 份报告,人工 ground-truth 标注),定期测我们 DeepSeek 判官的人工一致性。当一致性 < 阈值时触发判官重训练或换型,避免漂移。这条工程化校准链在我们当前的 7-pillar 体系下完全没有,加进来代价低 (人工 30-50 份,可在下一篇论文 supplement 里做)。

MAUVE / 分布距离作为 presentation 补充

MAUVE (NeurIPS 2021) 把生成文本与参考文本在 embedding 空间的分布散度作为单一文字质量指标,完全不需要逐句判官。如果我们已经为某些任务写了 reference report (上面 RACE 那一项),那 MAUVE 就是免费午餐:把报告与 reference 算个 MAUVE 数,作为 presentation 的确定性补充。Vendi Score、MTLD / MATTR 是同族指标,可作为词汇多样性的简单基线。

Rubric tree 二值聚合 (WildBench 范式)

AI2 的 WildBench v2 (2024) 与 InFoBench + STICK (2024) 都验证:把一个维度拆成 5-10 个二值子问 然后聚合,比让 LLM 直接打 1-5 Likert 稳得多。我们的 checklist 21 项已经是这个范式,但 analysis_depth (10 项混合) 和 presentation (12 项混合) 的 Tier B 还有 LLM 直接给单个判决的部分,可以拆得更细。STICK (self-augmented) 报告 +6.5% 一致性,几乎是免费收益。

这六项的整体策略

不是六项全做。优先级:判官族多样化 > 参考报告锚定 > 失败模式分类法 > 其余。判官族多样化是最低悬果,直接换二号判官跑一遍就能给所有 LLM 判官维度加误差棒;参考报告锚定 + DEFT 失败分类法是 v5 论文阶段值得专门写一节的两项工程,做了以后我们就直接对标 DeepResearch Bench RACE 与 FINDER 这两个 2025 末的 SOTA 工作,而不只是单一的 ALCE-style citation grounding。

如果评委问「为什么不做 X (atomic factuality / 内部一致性 / 视角)?」

第一层回答:承认确实没做。当前 7 pillar 集中在「引文是否真实」与「任务完成度 + 文字质量」,没有专门一项测原子事实正确性、内部一致性、视角多元、信源多样性。这是 v2 的边界。

第二层回答:这并不是疏忽,而是「先做什么」的优先级选择。真实性门反转 (§11.1) 是我们想优先证实的方法学命题,加性 vs 乘性门的对比需要的最小指标集就是 v2 那 4 项。在这个命题被证实之前先把指标体系铺到 10 项以上,会让结论解读变成「哪一项导致了排名变化」的多元归因问题,把核心方法学发现稀释掉。先证 v2 简单命题,再扩到 v4 复杂体系,是方法学清晰性的选择,不是技术上的疏忽。

第三层回答:v4 已经实施并跑过实验,见 §15.8。4 个新 pillar (source_diversity、perspective_balance、factual_exactness、internal_consistency) 全部落地为 Python verifier + 接入 score_deep_answer + 计算 composite_v4,跑了 25 份真实报告 (9 个 agent × 3 个任务)。结果是:头部 (camel-ai) 与尾部 (gpt-researcher / langchain-odr) 在 v4 下不变,但中部三个 agent 移动 ≥ 2 档 (deerflow +2、storm +4 effective、ii-researcher -2),且新维度独立地确认了 gpt-researcher 的「URL 真但论断错」失败模式 (其 factual_exactness 全部 = 0)。

15.8 v4 落地实验:在真实 DR 报告上的真实数字

v4 不只是纸面设计。4 个新 verifier 全部落地为生产 Python 代码 (src/verifiers/source_diversity_verifier.py、perspective_balance_verifier.py、factual_exactness_verifier.py、internal_consistency_verifier.py),接入 scripts/score_deep_answer.py,V4 Pro 路由通过 judge_client.call_judge_heavy() 走 ds_proxy。然后在已经跑过的真实 DR 报告上做了一次端到端的实验。

实验设计

  • 样本:9 个头部 agent × 3 个代表性任务 (dr_cross_deep_0001 / 0005 / 0010) = 27 份报告。这些报告之前都已经在 v2/v3 上打过分,所以可以做严格的 v2 → v4 对照。

区分度问题及 v4b 改进

第一次跑出的 v4 结果暴露了一个真实问题:中部 4 个 agent (smolagents / deerflow / storm / ii-researcher) 挤在 50 Elo 内,最差相邻 gap 只有 0.9 Elo (smolagents vs storm) 与 3.8 Elo,意味着排行榜在这个区间几乎不能区分。诊断原因有两条:

  • internal_consistency 饱和。跨 agent 跨度只有 0.87-1.00,标准差 ≈ 0.05。它占 13% 权重,但 8 个 agent 里有 7 个都在 0.95 以上,实际上等于给每份报告加一个常数,完全不区分。
  • source_diversity 严重低估。跨 agent 跨度 0.28-0.93,标准差 ≈ 0.21,是区分度最高的新 pillar 之一,但只给 6% 权重。

两轮改进各做了什么

① composite_v4b (无新 LLM 调用)。同样的 4 个 verifier、同样的可达性乘性门,只改两件事:

  • IC 数值缩放:原始 [0.85, 0.95] 线性映射到 [0, 1],把饱和区里的小差别放大成完整 1 分。
  • 权重重分配:IC 0.13 → 0.07,SD 0.06 → 0.12,把权重从信号弱的 pillar 移到信号强的 pillar。

② 扩样本。把任务从 3 个 (25 行) → 6 个 (44 行) → 10 个 (70 行),battles 从 46 涨到 109。每个 agent 平均 7-10 份报告参与排名。

③ composite_v4c (z-score 方差预算) ── 让饱和问题自动消失

v4b 的两项改动 (IC 重缩放 + 权重重分配) 是手工对症下药:我们观察到 IC 饱和,所以手工压它;观察到 SD 区分度高,所以手工放它。这有两个缺点 ── 一是只解决了我们注意到的那个 pillar,二是 0.13→0.07、0.06→0.12 这种重分配本身是个拍脑袋的数字。v4c 把同一件事做得有原则:

  1. 对每个 pillar i,在整个样本(全部 44 行)上计算总体均值 μ_i 和标准差 σ_i。
  2. 把每份报告的原始 pillar 值标准化为 z 分数: z_i = (p_i − μ_i) / σ_i。
  3. 用 sigmoid (sharpness=1.2) 把 z 分数压回 [0, 1]: sigmoid(1.2 · z)。
  4. 再用 v4b 同样的权重向量加权求和 ── 不重新调权重,只换数值变换层。

这一变换的几何意义:饱和 pillar (σ 接近 0) 的 z 分数对所有人都接近 0 → sigmoid(0) = 0.5,变成对所有人的恒定贡献,不再影响排名。高方差 pillar (σ 大) 的 z 分数仍跨越 [-3, +3] → sigmoid 输出仍接近 [0, 1],自动获得完整的区分能力。无需手工 _rescale_ic,无需重新调权重:variance budgeting 自己处理一切。

这是金融复合指数 (S&P MAPI、SDG 指数) 与多准则决策 (MCDM) 里的标准做法。代价是 v4c 不再是单份报告的纯函数 ── 它依赖样本统计量 μ, σ,所以每次重算排行榜要用同一个样本的 μ, σ 快照。我们把 μ, σ 一并写入 leaderboard_deep_v4c.json 以便复现。
